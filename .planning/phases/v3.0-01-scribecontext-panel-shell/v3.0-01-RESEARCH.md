# Phase v3.0-01: ScribeContext + Panel Shell — Research

**Researched:** 2026-03-11
**Domain:** React context + CSS flex layout + keyboard shortcut detection
**Confidence:** HIGH

## Summary

Phase v3.0-01 adds a ScribeContext provider to centralize Scribe state, a side panel shell that renders as a flex sibling to the OO editor iframe, a 2-button floating zone, and keyboard shortcuts (Ctrl+Shift+I double-tap). The codebase already has a working proof-of-concept for the flex sibling layout pattern (`OnlyOfficeAIAssistantPanel.tsx` renders conditionally as a flex sibling to `#onlyOfficeEditor`), an established portal-based floating button (`ScribeFloatingButton.jsx`), and a context provider pattern (`OnlyOfficeProvider.jsx`).

All required UI components exist in cozy-ui (Paper, Typography, IconButton) and no new dependencies are needed. The primary risk is OO iframe resize behavior -- the existing flex layout works because OO's `#onlyOfficeEditor` div is a flex child without explicit width, so it should shrink when a sibling appears. The `OnlyOfficeAIAssistantPanel` at 30% width already proves this works. The main engineering tasks are: (1) creating ScribeContext, (2) building the panel shell, (3) refactoring the floating button into a 2-button stack, (4) wiring Ctrl+Shift+I double-tap in plugin code.js, and (5) implementing popover/panel coexistence logic.

**Primary recommendation:** Follow the existing `OnlyOfficeProvider` pattern for ScribeContext, render the panel as a flex sibling in `View.jsx` (replacing the current `OnlyOfficeAIAssistantPanel` position), and use CSS transitions for the slide-in animation.

### Pattern 1: ScribeContext Provider
**What:** React context centralizing panel state + Scribe mode state
**When to use:** Any component that needs to read/write panel open/closed, or check if panel mode is active

**Recommended state shape:**
```javascript
// src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx
const ScribeContext = createContext()

const ScribeProvider = ({ children }) => {
  const [isPanelOpen, setIsPanelOpen] = useState(() => {
    try {
      return localStorage.getItem('scribe-panel-open') === 'true'
    } catch { return false }
  })

  // Persist to localStorage on change
  useEffect(() => {
    try {
      localStorage.setItem('scribe-panel-open', String(isPanelOpen))
    } catch { /* ignore */ }
  }, [isPanelOpen])

  const togglePanel = useCallback(() => {
    setIsPanelOpen(prev => !prev)
  }, [])

  const openPanel = useCallback(() => setIsPanelOpen(true), [])
  const closePanel = useCallback(() => setIsPanelOpen(false), [])

  const value = useMemo(() => ({
    isPanelOpen,
    togglePanel,
    openPanel,
    closePanel
  }), [isPanelOpen, togglePanel, openPanel, closePanel])

  return (
    <ScribeContext.Provider value={value}>
      {children}
    </ScribeContext.Provider>
  )
}

const useScribe = () => useContext(ScribeContext)
```

### Pattern 2: Flex Sibling Panel Layout
**What:** Panel renders as a conditional flex sibling next to `#onlyOfficeEditor`
**When to use:** Panel open state is true

**Key insight:** The existing `<div className="u-flex u-flex-grow-1">` in View.jsx already contains `#onlyOfficeEditor` as a flex child. The OO iframe fills this div. When a sibling with fixed width (400px) appears, the flex container shrinks the editor div automatically. `OnlyOfficeAIAssistantPanel` at 30% proves this.

```jsx
// In View.jsx, inside the u-flex div:
<div className="u-flex u-flex-grow-1">
  <div id="onlyOfficeEditor" style={{ flex: '1 1 auto', minWidth: 0 }} />
  <OnlyOfficeAIAssistantPanel />
  {isPanelOpen && <ScribePanel />}
</div>
```

The `#onlyOfficeEditor` div may need an explicit `flex: 1 1 auto` and `minWidth: 0` to ensure it shrinks properly. Currently it has no flex properties -- it relies on default flex behavior. Test this during implementation.

### Pattern 3: Slide-in Animation with CSS Transitions
**What:** Panel slides in from right with a smooth transition
**When to use:** Panel open/close toggle

```jsx
// Render panel always but control width/visibility via CSS
// OR render conditionally with CSS animation class
const panelStyle = {
  width: 400,
  flexShrink: 0,
  transition: 'width 200ms ease-out, opacity 200ms ease-out',
  overflow: 'hidden',
  boxShadow: '-2px 0 8px rgba(0, 0, 0, 0.1)'
}
```

**Recommendation:** Render the panel conditionally (not always-present with width:0) because CSS transitions on mount/unmount are tricky. Instead, use a simple approach: render when `isPanelOpen` is true, and let the flex layout handle the resize. The OO editor will reflow. If smooth animation is needed, render the panel always but toggle its width between 0 and 400px.

### Pattern 4: Double-Tap Detection (Ctrl+Shift+I)
**What:** Detect two Ctrl+Shift+I presses within 500ms in plugin code.js
**When to use:** Plugin keyboard shortcut handler (ES5 syntax required)

```javascript
// In plugin code.js (ES5 only!)
var lastCtrlShiftITime = 0;
var DOUBLE_TAP_MS = 500;

// Register on parent document (same pattern as existing Ctrl+I)
try {
  window.parent.document.addEventListener("keydown", function(e) {
    var isCtrlShiftI = (e.ctrlKey || e.metaKey) && e.shiftKey && e.key === "I";
    if (!isCtrlShiftI) return;
    e.preventDefault();

    var now = Date.now();
    if (now - lastCtrlShiftITime < DOUBLE_TAP_MS) {
      // Double-tap: toggle panel
      log("Ctrl+Shift+I double-tap: toggle panel");
      castIntent("TOGGLE_SCRIBE_PANEL", {}, true);
      lastCtrlShiftITime = 0; // Reset to prevent triple-tap
    } else {
      lastCtrlShiftITime = now;
    }
  });
} catch (e) {
  // Cross-origin fallback
}
```

**Note:** The first Ctrl+Shift+I press does nothing visible (it starts the timer). Only the second press within 500ms triggers the panel toggle. This is by design -- single Ctrl+Shift+I from the popover context is handled separately (see coexistence logic).

### Pattern 5: Popover Coexistence
**What:** When panel is open, inline Scribe redirects to panel; when closed, original behavior
**When to use:** useCozyBridge intent routing

The coexistence logic lives in View.jsx:
- `showScribeButton` state from useCozyBridge controls floating zone visibility
- When `isPanelOpen` is true: hide floating zone entirely, suppress `ScribePopover` open
- When `isPanelOpen` is true and Ctrl+I fires: instead of opening popover, the selected text context goes to the panel (Phase v3.0-02 will implement the actual chat, but the routing must be set up now)
- If popover is open and user opens panel: call `handleCancel()` on popover, then `openPanel()`

### Anti-Patterns to Avoid
- **Don't use transform: translateX for slide-in:** It would overlay rather than push the editor. Use width transition within the flex container so the editor actually resizes.
- **Don't add explicit width to #onlyOfficeEditor:** It currently relies on flex auto-sizing. Adding a width could break the existing layout. Instead, ensure it has `flex: 1 1 auto`.
- **Don't register keyboard shortcuts on the Cozy Drive window directly:** The plugin code.js registers on `window.parent.document` (OO editor iframe's parent). Cozy Drive should only handle intents received via postMessage.
- **Don't persist panel state on the server:** localStorage is appropriate for UI preferences. No cozy-stack API calls needed.

## Common Pitfalls

### Pitfall 1: OO Iframe Not Resizing
**What goes wrong:** Adding a 400px sibling might not shrink the OO editor iframe if the iframe has a fixed width or the parent flex container doesn't propagate correctly.
**Why it happens:** The `#onlyOfficeEditor` div has no explicit flex properties. OO creates its iframe inside this div with `height: 100%` (set by `forceIframeHeight`).
**How to avoid:** Add `style={{ flex: '1 1 auto', minWidth: 0 }}` to the `#onlyOfficeEditor` div. The `minWidth: 0` is critical -- without it, flex items won't shrink below their content size. The iframe width should follow the container div width. Test empirically: open panel, verify iframe actually narrows.
**Warning signs:** Panel overlaps editor instead of pushing it, or editor doesn't shrink at all.

### Pitfall 2: ScribeContext Provider Placement
**What goes wrong:** If ScribeContext wraps too low in the tree, components above it can't access panel state. If too high, unnecessary re-renders.
**Why it happens:** View.jsx needs ScribeContext (for panel rendering), but the Toolbar might also need it (future phases).
**How to avoid:** Place ScribeProvider in `index.jsx` wrapping `<Editor />`, inside `OnlyOfficeProvider`. This gives access to both Editor/View and any future toolbar integration.

### Pitfall 3: Floating Zone Z-Index Conflicts
**What goes wrong:** Two buttons in the floating zone overlap with OO dropdowns or other UI elements.
**Why it happens:** OO uses very high z-index values for its UI. The current floating button uses z-index 100000.
**How to avoid:** Keep z-index 100000 for the floating zone container. Both buttons inherit it. The portal on `document.body` already avoids stacking context issues.

### Pitfall 4: CSS Transition on Conditional Render
**What goes wrong:** Panel appears instantly without animation because React mounts it already at full width.
**Why it happens:** CSS transitions require a starting state to transition FROM. If the element is mounted at its final width, no transition occurs.
**How to avoid:** Two options: (a) always render the panel but toggle its width between 0 and 400px (simpler CSS transitions but always in DOM), or (b) use a two-step mount: render at width 0, then requestAnimationFrame to width 400px. Option (a) is recommended.

### Pitfall 5: Ctrl+Shift+I Conflicts with Browser DevTools
**What goes wrong:** Ctrl+Shift+I is the browser DevTools shortcut on Windows/Linux.
**Why it happens:** Browser intercepts the shortcut before JavaScript keydown event fires.
**How to avoid:** The shortcut registration happens on `window.parent.document` inside the OO iframe, and `e.preventDefault()` is called. In practice, OO iframes may not have focus when the user presses the shortcut. The double-tap requirement (two presses within 500ms) makes accidental DevTools triggers less likely -- the first press would open DevTools, and the user would need to press again quickly. **This is a known tradeoff documented in CONTEXT.md.** If this proves problematic in practice, a different shortcut can be chosen in a future phase. For now, implement as designed.

### Pitfall 6: localStorage SSR Safety
**What goes wrong:** `localStorage.getItem` throws during SSR or in restrictive iframe sandboxes.
**Why it happens:** Some environments don't have localStorage available.
**How to avoid:** Wrap in try/catch with fallback to `false`. The existing codebase accesses localStorage directly in other places, so this is an established pattern.

## Open Questions

1. **OO Editor Iframe Resize Behavior**
   - What we know: OnlyOfficeAIAssistantPanel at 30% works as flex sibling. The `#onlyOfficeEditor` div has no explicit flex properties. OO creates iframe with `height: 100%` via `forceIframeHeight`.
   - What's unclear: Whether the iframe width follows its container div width automatically, or if OO sets an explicit width on the iframe. Also, whether OO re-renders its toolbar/canvas when the iframe resizes.
   - Recommendation: First task should be a manual test -- add `minWidth: 0` and `flex: '1 1 auto'` to `#onlyOfficeEditor`, conditionally render a 400px div, and verify OO editor actually shrinks and remains functional. This is the go/no-go gate mentioned in STATE.md.

2. **cozy-ui Icon Availability for Close Button**
   - What we know: cozy-ui has `Icon` component and icon set. `IconButton` is available.
   - What's unclear: Exact icon name for "close/X" in cozy-ui's icon set (might be `cross`, `close`, or require importing from a specific path).
   - Recommendation: Check cozy-ui icon set during implementation. Fallback: inline SVG X icon (same pattern as ScribeFloatingButton).

3. **OPEN_SCRIBE_PANEL Intent from Popover**
   - What we know: Ctrl+Shift+I from popover context should open panel with single press.
   - What's unclear: How to detect "from popover context." The popover is a React component; it can call `openPanel()` directly via ScribeContext.
   - Recommendation: Add a keyboard shortcut handler inside ScribePopover that listens for Ctrl+Shift+I and calls `openPanel()` + `onCancel()`. No intent needed -- it's all within React.
