<objective>
Add post-injection selection so the user sees injected content highlighted after Replace or Insert.

Purpose: After Scribe injects AI-modified text, the user has no visual feedback about what was injected. Selecting the injected content gives immediate confirmation. This uses a sentinel marker strategy: unique marker strings are inserted flanking the content, then a second callCommand finds them via doc.Search(), selects the range between them, and removes the markers.

Output: After injection, the injected content is highlighted in the OO editor. If the sentinel approach fails, content is still injected (graceful degradation).
</objective>

This task adds sentinel markers to the injected content and a second callCommand that finds them, selects the content between them, and removes them. ALL code must be ES5 (var, no arrow functions, no template literals, no const/let).

**Part A: Sentinel marker constants and Asc.scope setup**

At the top of the buildAndInject function (outside the callCommand, around line 180-190), before calling callCommand:
1. Generate unique sentinel strings: `var sentinelStart = "SCBS" + Date.now(); var sentinelEnd = "SCBE" + Date.now();`
2. Pass via Asc.scope: `Asc.scope._sentinelStart = sentinelStart; Asc.scope._sentinelEnd = sentinelEnd;`

**Part B: Add sentinel runs to content array inside callCommand #1**

Inside the existing callCommand function body, AFTER the content-building loop and space run logic, BEFORE `doc.InsertContent(content)`:

1. Read sentinels from scope: `var sentinelStart = Asc.scope._sentinelStart; var sentinelEnd = Asc.scope._sentinelEnd;`
2. Add start sentinel as FIRST run of first paragraph:
   - `var startRun = Api.CreateRun(); startRun.AddText(sentinelStart); startRun.SetFontSize(1);`
   - Since AddElement only appends, and we need the sentinel BEFORE existing runs: modify the content-building loop. When building the first paragraph (i === 0), add the sentinel run via `p.AddElement(startRun)` BEFORE adding content runs. This integrates cleanly with the existing loop.
   - If needSpaceBefore is true (from 20-01), the order in the first paragraph should be: [sentinel run] [space run] [content runs]. The sentinel marks the START of injected content. The space IS part of the injected content.
3. Add end sentinel as LAST run of last paragraph:
   - `var endRun = Api.CreateRun(); endRun.AddText(sentinelEnd); endRun.SetFontSize(1);`
   - `content[content.length - 1].AddElement(endRun);`
   - If needSpaceAfter added a space run to the last paragraph, the sentinel should come AFTER the space (so the space is included in the selection). Append sentinel after the space run.

**Part C: selectInjectedContent function (callCommand #2)**

Create a new function `selectInjectedContent(sentinelStart, sentinelEnd)` that:

1. Passes sentinels via Asc.scope: `Asc.scope._sentinelStart = sentinelStart; Asc.scope._sentinelEnd = sentinelEnd;`
2. Calls `window.Asc.plugin.callCommand(function() { ... }, false, false, callback)` with:
   - Read sentinels: `var sStart = Asc.scope._sentinelStart; var sEnd = Asc.scope._sentinelEnd;`
   - `var doc = Api.GetDocument();`
   - Search: `var startResults = doc.Search(sStart, true); var endResults = doc.Search(sEnd, true);`
   - If both found (length > 0):
     a. Get positions: `var contentStart = startResults[0].GetStartPos(); var contentEnd = endResults[0].GetEndPos();`
     b. Remove sentinels via SearchAndReplace: `doc.SearchAndReplace({searchString: sStart, replaceString: ""}); doc.SearchAndReplace({searchString: sEnd, replaceString: ""});`
     c. Adjust positions after removal: start sentinel was at contentStart, it's removed so all positions shift left by sStart.length. End sentinel was at (contentEnd - sStart.length - sEnd.length) after start removal.
     d. Compute final range: `var adjStart = contentStart; var adjEnd = contentEnd - sStart.length - sEnd.length;`
     e. Select: `var selectRange = doc.GetRange(adjStart, adjEnd); if (selectRange) selectRange.Select();`
   - If search fails: log warning, skip selection (graceful degradation)
3. In the callback: `pasteInProgress = false;`

**IMPORTANT FALLBACK:** If position arithmetic proves unreliable (positions shift unpredictably across paragraphs), the executor should try the alternative approach from the research:
1. First try: remove sentinels then select adjusted range (as above)
2. If that doesn't work: select the range INCLUDING sentinels (contentStart to contentEnd), then remove sentinels. The SearchAndReplace may or may not deselect. Test and adapt.
3. If neither works: skip selection entirely. Log a warning. The content is still injected. This is graceful degradation.

**Part D: Wire callCommand #1 callback to callCommand #2**

In the existing buildAndInject callCommand callback (currently around line 356-361):
1. Do NOT set `pasteInProgress = false` here anymore -- that moves to selectInjectedContent's callback
2. Instead, call `selectInjectedContent(sentinelStart, sentinelEnd)` -- the sentinelStart/sentinelEnd variables are in closure scope from the outer buildAndInject function
3. Keep `callbackFired = true` and `clearTimeout(fallbackTimer)` in callCommand #1's callback
4. Add a safety timeout: if selectInjectedContent doesn't complete within 3 seconds, set `pasteInProgress = false` anyway (prevents permanent lock)

**Part E: Handle PasteHtml fallback path**

If buildAndInject falls back to pasteHtml (fallbackTimer fires or catch block), pasteInProgress is already handled by pasteHtml. No sentinel cleanup needed since InsertContent never ran.

**Anti-patterns to avoid (from research):**
- Do NOT try to Search in the same callCommand as InsertContent (content not committed yet)
- Do NOT use invisible Unicode characters (ZWSP) as sentinels -- OO may strip them
- Do NOT skip the safety timeout on callCommand #2 -- if Search fails, pasteInProgress must still reset
