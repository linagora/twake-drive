# Phase v3.2-01 — UI Design Contract

> Visual and interaction contract for the discreet « Inclure » zone (3 checkboxes above the Scribe prompt). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **SCOPE GUARD (HARD):** This phase is **state-only, presentational**. The checkbox state lives in the panel and is read by NO LLM call. Do NOT touch `sendMessage` / `callScribeAIWithReask` / `encodeSelectionForPrompt` / prompt assembly. Injecting any context into the prompt is v3.2-02 (discussion+selection) and v3.2-03 (document). The executor must NOT add any context value to the prompt in this phase — the LLM call behavior is byte-for-byte identical to v3.1.

| Property | Value |
|----------|-------|
| Tool | none (cozy-ui + MUI; NOT shadcn) |

**Controls — exact components (matching live usage):**

- **Checkbox:** `cozy-ui/transpiled/react/Checkbox` with `size="small"` and `style={{ padding: 2 }}` — this is the EXACT control + density already used for the discreet dev-panel toggles in `ScribeResultPanel.jsx` (lines 946–951). Do NOT introduce a custom toggle. The whole checkbox+label is wrapped in a `<label>` for hit-area + association (same pattern as the dev toggles, lines 933–956).
- **Label text:** a `<span>` styled `color: theme.palette.text.secondary` (muted) — identical to the dev-toggle label span (line 952). Font size 12px (one notch above the 10px dev micro-labels, because this is a user-facing control rather than a developer affordance).
- **Section caption « Inclure »:** a muted micro-caption — `cozy-ui Typography variant="caption" color="textSecondary"` OR a `<span>` at `fontSize: 11`, `text.secondary`, `textTransform: 'uppercase'`, `letterSpacing: 0.5px` (the dev-label treatment, lines 894–902). Discretion: keep it subtle; it must read as a quiet section header, never a heading that competes with the prompt.

**Theme:** read `useTheme()` from `cozy-ui/transpiled/react/styles` (already imported in every Scribe component). Use `theme.palette.*` tokens only — NO hard-coded light/dark hex for text or surfaces. Dark detection, where needed, follows the established idiom `(theme.palette.type || theme.palette.mode) === 'dark'`.

## Spacing Scale

Declared values (multiples of 4; consistent with the 4/8/12 rhythm already in ChatInput/SelectionChip):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Caption→row gap; vertical gap between « Inclure » caption and the checkbox row; checkbox internal `padding: 2` (control-intrinsic, see exception) |
| sm | 8px | Horizontal gap between checkboxes in the row; gap before the SelectionChip when it appears |
| md | 12px | Horizontal padding of the zone, aligning with ChatInput's `padding: '8px 12px'` so the zone and input share a left edge |
| lg | 16px | (not used in this compact zone) |

**Zone-specific layout values:**

- Zone container padding: `8px 12px` top/bottom + left/right — mirrors `ChatInput` (line 156) so the « Inclure » zone sits flush above the input with the same horizontal inset.
- Caption → checkbox-row vertical gap: `4px`.
- Inter-checkbox gap: `8px` (`display: flex; gap: 8px`).
- Visual separation **from the prompt input below:** NONE added by this zone — the existing `borderTop` on ChatInput already draws the divider; the « Inclure » zone renders ABOVE that border, so the prompt keeps its own top divider and the zone reads as a distinct band.
- Visual separation **from the fragment/message cards above:** the zone is part of the input footer cluster (renders inside/just above `ChatInput`, below the scrollable message list), so the existing scroll boundary separates it from cards. Do NOT add a second border that would make the zone "heavy."

**Exceptions:** cozy-ui `Checkbox size="small"` carries its own intrinsic control padding; we pass `style={{ padding: 2 }}` to tighten it to the dev-toggle density. The 2px is a control-internal value, not a layout token, and is the only non-4-multiple permitted (it matches the shipped dev-toggle pattern).

3 roles (well within the 3–4 limit):

| Role | Size | Weight | Line Height | Color |
|------|------|--------|-------------|-------|

| (existing) Prompt placeholder / body | 14px | 400 | 20px (1.43) | unchanged — owned by ChatInput, listed only for contrast reference |

Weights: exactly **2** — regular (400) + semibold (600). The caption is the only semibold element; all checkbox labels are regular and muted so the zone stays quiet.

**Discretion rationale (CTX-UX-05, HARD):** every text element in the zone uses `text.secondary` (muted), is ≤12px, and the controls are `size="small"`. The zone's largest type (11px caption) is smaller than the 14px prompt body and the 12px SelectionChip italic — guaranteeing the zone never out-weights the input it sits above.

## Layout & Interaction Contract

**Placement (locked CTX-UX / ROADMAP SC#1):** the « Inclure » zone renders in the side panel **directly above the prompt input**, inside the input footer cluster. Concretely: at the top of `ChatInput`'s footer container, ABOVE both the `SelectionChip` render slot and the textarea row. Order top→bottom inside the footer: `[ « Inclure » caption + checkbox row ]` → `[ SelectionChip (when sélection checked) ]` → `[ textarea + send button ]`.

**Structure:**
```
Inclure                      ← muted caption (11px, uppercase, semibold, text.secondary)
[☐ document]  [☐ discussion]  [☑ sélection]   ← row, gap 8px, labels 12px muted
```

**Row vs wrap (Claude's discretion → DECIDED):** the checkbox row uses `display: flex; flex-wrap: wrap; gap: 8px; row-gap: 4px`. With short labels (document/discussion/sélection) the three fit on one row at the default 400px panel width; on a narrowed panel (min 280px per `setPanelWidth` clamp) they wrap gracefully to a second line. Wrapping is preferred over horizontal scroll or truncation.

**The three checkboxes (locked CTX-UX-01..03 / SC#2):**

| Checkbox | i18n label key | Default | Conditional? |
|----------|----------------|---------|--------------|
| document | `Scribe.include.document` | **unchecked** | always rendered |
| discussion | `Scribe.include.discussion` | **unchecked** | always rendered |
| sélection | `Scribe.include.selection` | **checked** | rendered ONLY when `currentSelection != null` |

**Conditional « sélection » behavior (locked CTX-UX-04 / SC#3):**
- The « sélection » checkbox **mounts only when `currentSelection != null`** (read from `ScribeContext`). When no selection exists, the checkbox is absent (not disabled, not greyed — absent), keeping the row to two items.
- On the transition `null → selection`, the « sélection » checkbox appears **checked by default** (opt-OUT model: a fresh selection is included until the user unchecks it).
- **Checked → SelectionChip visible; unchecked → SelectionChip hidden.** The chip is the existing `SelectionChip.jsx`, unchanged.
- **State plumbing (planner's call, per CONTEXT DESIGN NOTE):** prefer a NEW lightweight "selection included" boolean over reusing `dismissSelection()`. Unchecking must hide the chip WITHOUT wiping the underlying `currentSelection` — so re-checking the box restores the same chip. The existing `selectionDismissedRef` "dismiss this exact text" behavior (the chip's × button) stays a SEPARATE concern from the checkbox. The UI-SPEC fixes only the visible behavior; the exact state location (ScribeContext vs panel-local) is a plan-level decision (SC#5 requires the three booleans live in shared/panel state ready for v3.2-02/03 to read). **Disqualified:** a component-local `useState` inside the « Inclure » zone — `sendMessage` (in ScribeContext) MUST be able to read these three booleans at the prompt-assembly seam in v3.2-02/03, so they must live in ScribeContext state (or a sibling context visible to `sendMessage`), not in a leaf component's local state.

**Visible transition (Claude's discretion → DECIDED):**
- SelectionChip show/hide on checkbox toggle: a quick **opacity + height collapse, 150ms ease** (matches the 150ms transitions already used on the send button and resize handle). Acceptable simpler fallback: instant show/hide with no animation — discretion permits either, but if animated, cap at 150ms so it never feels heavy. Respect `prefers-reduced-motion` (the panel already gates animation on it in `ScribePromptInput`): under reduced motion, snap with no transition.
- The « sélection » checkbox appearing/disappearing itself: no entrance animation required; mount/unmount is acceptable. (Animating a checkbox in/out risks layout jank in the compact row.)

**Static-state guarantee (SC#5):** toggling any of the three checkboxes changes ONLY local/shared boolean state (and, for « sélection », chip visibility). It produces NO change to what is sent to the LLM. The executor must add a code comment at the state seam noting "read by v3.2-02/03; no prompt injection in v3.2-01."

## Accessibility

- **Label association:** each checkbox + its text are wrapped in a `<label>` (the shipped dev-toggle pattern), giving an implicit label and a full-width click target. Equivalent: `<Checkbox>` with an `aria-label` resolved from the same `t()` key. Use the `<label>` wrap to match existing code.
- **Section caption association:** group the three checkboxes in a container with `role="group"` and `aria-label={t('Scribe.include.label')}` so screen readers announce the « Inclure » grouping. The visible « Inclure » caption is the human-visible echo of that group label.
- **Keyboard reachability & order:** checkboxes are natively focusable. Tab order follows DOM order: they sit in the input footer, BEFORE the textarea and send button, and AFTER the scrollable message thread. They must NOT be inserted into the thread's roving-keyboard navigation (the `onArrowUp`/thread controller in `ChatInput`/`ChatMessageList`) — they are plain Tab-reachable form controls, not roving-nav cards. Confirm Tab from the last message → checkboxes → textarea → send forms a sensible linear order.
- **Conditional checkbox appearing/disappearing (aria):** wrap the « sélection » slot in a container with `aria-live="polite"` so its appearance (when a selection arrives) is announced without stealing focus. **The `aria-live` wrapper div is ALWAYS mounted** (it renders empty / `null` children when `currentSelection == null`); only the inner `<label>`+`<Checkbox>` for « sélection » conditionally mounts — this keeps the live-region registration stable across selection appearing/disappearing. Do NOT move focus to the newly-appeared checkbox (CTX: opening the panel with text selected should show it checked, not grab focus from the prompt — `ChatInput` deliberately focuses the textarea on open). On disappearance (selection cleared), no announcement beyond the live region's natural update.
- **Checked-state semantics:** rely on native checkbox `checked`/`aria-checked`; no custom ARIA needed.
- **Contrast:** `text.secondary` on the panel surface meets contrast in both themes (cozy-ui palette guarantees this); do not drop opacity below the palette token.
