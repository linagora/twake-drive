<domain>
## Phase Boundary

First phase of milestone v3.2 « Contexte enrichi du prompt ». Delivers the **static UI only**: a discreet « Inclure » zone above the prompt in the Scribe side panel, with three checkboxes (« document », « discussion », « sélection ») whose state is held in the panel but **NOT yet injected into any LLM call**. The actual prompt wiring is explicitly v3.2-02 (discussion+selection) and v3.2-03 (document) — out of scope here.

Covers requirements CTX-UX-01..05.

**Out of scope (do NOT build here):** any change to `sendMessage`/`callScribeAIWithReask`/prompt assembly; any docx-document extraction; the conversation-history or selection injection into the prompt. This phase is purely presentational + local checkbox state.
</domain>

<decisions>
## Implementation Decisions

### Placement & structure (locked)
- The « Inclure » zone renders in the side panel **directly above the prompt input** (`ChatInput.jsx` / `ScribePromptInput.jsx`), inside the Scribe panel surface (`ScribeResultPanel.jsx` chat layout).
- A short section label « Inclure » (i18n) followed by the three checkboxes, laid out compactly (inline/row if space allows).

### The three checkboxes (locked — CTX-UX-01..03)
- « document » — unchecked by default.
- « discussion » — unchecked by default.
- « sélection » — **conditional**: rendered ONLY when an active selection exists (`currentSelection != null` in `ScribeContext`); **checked by default** as soon as a selection exists.

### Selection coupling (locked — CTX-UX-04)
- When « sélection » is checked, the existing selection display (`SelectionChip.jsx`) is shown. When unchecked, the chip is hidden.
- DESIGN NOTE for the planner (not for the UI-SPEC to over-fix): prefer a NEW lightweight "selection included" boolean over reusing `dismissSelection()` — unchecking should hide the chip WITHOUT wiping the underlying `currentSelection` (so re-checking restores it, and the existing `selectionDismissedRef` dismiss-this-exact-text behavior stays separate). The UI-SPEC should describe the visible behavior (checkbox toggles chip visibility); the exact state plumbing is a plan-level call.

### Discreet styling (locked — CTX-UX-05, HARD constraint)
- The checkboxes must be **visually discreet** — small control size, muted/secondary text, light integration above the prompt; they must NOT compete visually with the prompt input or the fragment cards.
- **Theme-aware** (light/dark) via the existing panel theming (`useTheme()` + `scribe.styl`), consistent with the rest of the Scribe panel.
- Reuse the existing Scribe/cozy-ui design system (MUI Checkbox or cozy-ui equivalent already used elsewhere in the panel) rather than introducing a new control style.

### i18n (locked — consistency with v3.1-07, just shipped)
- All new labels (« Inclure », « document », « discussion », « sélection ») go through `t()` with a new `Scribe.include.*` key group, translated in the **5 target locales fr/en/de/es/it** — **zero hard-coded strings** (the v3.1-07 literal-audit gate and parity gate are now in place; honor them).

### Claude's Discretion (UI researcher decides)
- Exact control (MUI `Checkbox` size="small" vs a custom compact toggle), label typography scale, spacing tokens, and whether the row wraps on narrow panels.
- Whether the section label is a visible « Inclure » caption or a subtler treatment — but it must remain discreet.
</decisions>

<specifics>
## Specific Ideas

- Default-checked « sélection » when a selection exists means: opening the panel with text selected should immediately show the selection chip with its checkbox ticked — the user opts OUT, not in, for selection.
- « document » and « discussion » default-unchecked means enriching with those is an explicit opt-in (they can be large / change tone).
- Static phase: clicking the checkboxes changes local state + (for « sélection ») chip visibility, but produces NO change in what is sent to the LLM yet. A later phase reads this state at the prompt-assembly seam.
</specifics>
