## Summary

This is a **wiring phase**, not a greenfield one. Every integration point already exists and was deliberately left extensible by v3.2-02 (verified PASSED, 160/160). There are **no new third-party packages**. The work is four concrete pieces:

1. **A new full-document extraction path in the OO plugin (`code.js`, ES5).** It must reuse the existing per-element markdown emitter (`paragraphToMarkdown`, `extractTableCells`, …) but drive it over the whole document instead of the selection range. **The single most important finding of this research:** `doc.GetAllParagraphs()` does **NOT** include paragraphs inside table cells — this is verified by the codebase's own compensating code and comment at `code.js:638` (`// Also scan paragraphs inside table cells (GetAllParagraphs doesn't include them)`). A naive `GetRangeBySelect()→GetAllParagraphs()` swap to `doc.GetAllParagraphs()` will **silently drop all table content**. The robust, evidence-backed primitive for whole-document iteration is the document-order walk `doc.GetElementsCount()` / `doc.GetElement(i)` branching on `GetClassType()` (`"paragraph"` vs `"table"`) — exactly the pattern the plugin already uses successfully in `hookDumpState` (`code.js:3304-3322`).

2. **Host→plugin request/response plumbing.** The cozy-bridge intent protocol is one-directional (plugin emits intent → host responds) and is the wrong shape for "host asks plugin to extract on demand." The codebase already contains the right pattern: the dev-test channel (`{ scribeTest, reqId }` → `{ scribeTestResult, reqId, … }`, `code.js:3361-3376`). Mirror it with a dedicated message type. This **bypasses `validateIntent`'s 1 MB cap entirely** (that cap only gates `cozy-bridge:intent`/`:response`).

3. **Deterministic prompt composition at the existing seam.** `ScribeContext.sendMessage` is already `async`, already reads include flags live via refs, and already calls the flag-parameterized `buildChatSystemPrompt`. Add `includeDocumentRef`, await the extraction, prepend a `[Full document]…[End of full document]` block to the current user message, and grow `contextSourceFraming` with an `includeDocument` branch — without touching the frozen `RESPONSE_CONTRACT_CORE`/`CARDINALITY_CHAT`.

4. **Size strategy + never-silent feedback.** Budget via a `cozy-flags` numeric flag (default unset = unlimited), char-based truncation at a block boundary, and a per-turn `documentNotice` flag on the user message that `ChatMessageList` renders as the approved discreet notice (truncation) or `ErrorBubble` (extraction failure).

**Primary recommendation:** Reuse, don't rebuild. Drive the existing leaf emitter from a document-order `GetElement` walk (not `GetAllParagraphs`); plumb extraction over a new dev-test-style `reqId` channel wired through `View.jsx` into `ScribeContext` (mirror the `setPanelActions` pattern); gate everything on a new `includeDocumentRef`; bound size with a `cozy-flags` char budget (default unlimited) and surface truncation/failure through a `documentNotice` field on the user message.

### Pattern 1: Whole-document iteration via the document-order `GetElement` walk (NOT `GetAllParagraphs`)

**What:** Iterate top-level blocks in document order and dispatch to the existing leaf emitter.
**Why:** `doc.GetAllParagraphs()` excludes table-cell paragraphs (see Pitfall 1). The `GetElement` walk is the proven primitive (`hookDumpState`, `code.js:3304-3322`) and preserves paragraph/table ordering without the selection path's fragile position-overlap arithmetic.
**ES5-only.** All leaf helpers live INSIDE a `callCommand` closure (sandbox) — see Pitfall 4 for the reuse constraint.

```javascript
// Source: pattern derived from code.js:3304-3322 (hookDumpState) + reusing the
// selection emitter's leaf helpers (code.js:2417 getHeadingLevel, 2429 isListParagraph,
// 2447 getDrawingMarker, 2219 paragraphToMarkdown, 2619 extractTableCells).
// MUST be defined INSIDE the callCommand body (ES5 sandbox).
var doc = Api.GetDocument();
var mdParts = [];
var orderedCounters = {};
var blockCount = doc.GetElementsCount();
for (var i = 0; i < blockCount; i++) {
  var el = doc.GetElement(i);
  var ct = el.GetClassType ? el.GetClassType() : "";
  if (ct === "table") {
    mdParts.push({ md: "[TABLE:" + tableIndex + "]\n" + extractTableCells(el) + "\n[/TABLE]", isList: false });
    tableIndex = tableIndex + 1;
    continue;
  }
  if (ct !== "paragraph") continue;
  // images, headings, lists handled exactly as the selection path does, but with
  // clipStart=clipEnd=0 (no selection clipping for full-doc) — see code.js:2786-2877.
  var md = paragraphToMarkdown(el, 0, 0);
  // apply heading/list prefixes (reuse getHeadingLevel / isListParagraph / indentToLevel)
  mdParts.push({ md: md, isList: /* listInfo */ false });
}
```

> **Alternative (mirror the selection path):** keep `paragraphs = doc.GetAllParagraphs()` + `allTables = doc.GetAllTables()` with `selStart=0/selEnd=BIG`. This **requires** reworking the table-emission trigger because, with cell paragraphs absent, no top-level paragraph's `GetStartPos()` falls inside a table's range, so the existing `insideTable` branch never fires and tables are dropped. The `GetElement` walk avoids this trap entirely. Recommend the walk.

### Pattern 2: Host→plugin request/response over a dedicated `reqId` channel

**What:** A correlated round-trip mirroring the dev-test channel, NOT the intent protocol.
**Why:** Bridge intents are plugin→host; the host has no "ask plugin, await reply" verb. The dev-test channel already implements exactly this (`code.js:3361-3376`). A distinct message type also sidesteps `validateIntent`'s 1 MB cap (Pitfall 3).

Plugin side (`code.js`, ES5) — new listener, sibling of the dev-test one:
```javascript
// Source: pattern from code.js:3361-3376 (dev-test channel) + 1996-2021 (subscribe listener)
window.addEventListener("message", function(event) {
  var msg = event.data;
  if (!msg || msg.type !== "cozy-bridge:extract-document") return;
  var reqId = msg.reqId;
  extractFullDocumentToMarkdown(function(result) {     // runs the callCommand walk above
    postToAncestors({
      type: "cozy-bridge:document-extracted",
      reqId: reqId,
      md: result.md,                                   // may be large; not validateIntent-gated
      error: result.error || null
    });
  });
});
```

Host side — implement in `View.jsx` (it already owns `broadcastToFrames` and message listeners) and inject into `ScribeContext` via a setter mirroring `setPanelActions` (`View.jsx:238-242`, ScribeContext `panelActions`):
```javascript
// Source: pattern from View.jsx:86-101 (broadcastToFrames) + 121-132 (plugin-ready listener)
const extractFullDocument = useCallback(() => new Promise((resolve) => {
  const reqId = (crypto.randomUUID && crypto.randomUUID()) || String(Date.now())
  const onMsg = e => {
    const m = e.data
    if (!m || m.type !== 'cozy-bridge:document-extracted' || m.reqId !== reqId) return
    window.removeEventListener('message', onMsg)
    resolve({ md: m.md || '', error: m.error || null })
  }
  window.addEventListener('message', onMsg)
  broadcastToFrames({ type: 'cozy-bridge:extract-document', reqId })
  // recommend a timeout → resolve({ md:'', error:'timeout' }) so a dead plugin
  // yields the DEC-UI-03 "unavailable" notice instead of hanging the send.
}), [broadcastToFrames])
```
Then `setExtractFullDocument(extractFullDocument)` into ScribeContext (new setter), consumed inside `sendMessage`.

### Pattern 3: Inject the document block at the seam, gated live, ephemeral (not stored)

**What:** Prepend a named block to the **current** user message content; do **not** store it on the message object.
**Why:** D-05 = fresh each send. Unlike the selection (stored in `m.selection` and replayed in history), the document must be **ephemeral** — re-extracted each send, never persisted into the transcript (otherwise history replay would resend a stale doc and bloat every turn). Mirror `includeSelectionRef`/`includeDiscussionRef` with a new `includeDocumentRef` (ScribeContext currently does NOT mirror `includeDocument` — comment at `ScribeContext.jsx:85`).

```javascript
// In sendMessage, after the existing selectionIncluded computation:
const includeDocument = includeDocumentRef.current
let documentMd = ''
let documentNotice = null
if (includeDocument && extractFullDocumentRef.current) {
  const { md, error } = await extractFullDocumentRef.current()
  if (error) { documentNotice = 'unavailable' }            // DEC-UI-03
  else { const r = applyBudget(md); documentMd = r.md; if (r.truncated) documentNotice = 'truncated' } // DEC-UI-01
}
// recommended block ORDER in the current user content: document (broad) → selection (focus) → user text
const docBlock = documentMd ? `[Full document]\n${documentMd}\n[End of full document]\n\n` : ''
// ... selBlock as today ('[Selected text from document]…') ...
// current user content = docBlock + selBlock + text
// store documentNotice on the user message for ChatMessageList (NOT documentMd):
userMessage.documentNotice = documentNotice
```

### Pattern 4: Grow `contextSourceFraming` with a document branch (D-06)

**What:** Add ONE short, leading-space-prefixed English sentence when `includeDocument` is true, appended AFTER the contract block exactly like the v3.2-02 seed.
**Why:** Keeps the hard-gated `RESPONSE_CONTRACT_CORE`/`CARDINALITY_CHAT` byte-identical; the per-source sentences ARE the enumeration — do **not** build a literal "1./2./3." list and do **not** instruct the model to mention sources (Mistral-Small meta-commentary fragility).

```javascript
// Source: extend scribeAI.js:144-156 (contextSourceFraming). Add includeDocument to the flags object.
if (includeDocument) {
  out += ' The full document is provided below for reference; use what is relevant to the request and ignore the rest.'
}
// When BOTH document and selection are included, the selection-is-focus relationship
// is best expressed by ONE combined clause (D-04), e.g. appended only when both true:
//   ' The selection marks the user’s current focus within that document.'
```
`buildChatSystemPrompt(selectionMd, flags)` already threads a `flags` object (`scribeAI.js:172-181`) — add `includeDocument` to it. **Do NOT** pass the document markdown through `markerPreservationClauses` (those clauses protect a selection that may be transformed/re-injected; the document is read-only context — adding them would tell the model to "preserve markers" it is only reading, inviting echo/meta-commentary). [See Open Question 2.]

### Pattern 5: Size budget via `cozy-flags`, head-keeping block-boundary cut

**What:** `const budget = flag('drive.scribe.contextBudget')` (a number of characters; `null`/`undefined` ⇒ unlimited). If `documentMd.length > budget`, keep the **head** up to the budget, cut at the **last complete block boundary** (`\n\n` join produced by the emitter), set `truncated = true`.
**Why head-keeping:** (a) deterministic and simple; (b) the selection is already sent separately (D-04), so the user's focus is covered regardless of where the doc is cut; (c) title/headings/intro are front-loaded and carry the most structural context. Reserve nothing for "the answer" — the model's output budget is separate from its input budget. The UI copy is intentionally cut-point-agnostic ("only part of it") per UI-SPEC, so this choice doesn't constrain the strings.
**Optional token heuristic:** if the planner prefers a token budget, estimate `chars ≈ tokens × 4` (FR/EN average) — document the heuristic; do not ship a tokenizer (deferred).

### Pattern 6: Never-silent feedback (DEC-UI-01 / DEC-UI-03)

Store a per-turn `documentNotice` on the user message (`'truncated'` | `'unavailable'` | absent). `ChatMessageList.jsx` renders it in thread order right after the `UserBubble` (the `msg.role === 'user'` branch at `ChatMessageList.jsx:429-437`):
- `'truncated'` → a discreet `role="status"` row cloned from the loading indicator (`ChatMessageList.jsx:519-534`): `alignSelf:flex-start; display:flex; gap:8; padding:'8px 12px'; opacity:0.7; fontSize:13; color:text.secondary` + info `Icon` (14px) + `t('Scribe.include.documentTruncated')`. **No red, no accent.**
- `'unavailable'` → reuse `ErrorBubble` (`ChatMessageList.jsx:439-447`) with `t('Scribe.include.documentUnavailable')`.
- Default config (no budget) ⇒ never truncates ⇒ no notice (silent success is correct).

### Anti-Patterns to Avoid
- **Swapping `GetRangeBySelect()`→`doc.GetAllParagraphs()` and assuming tables come along.** They don't (Pitfall 1).
- **Routing the (potentially multi-MB) document over `cozy-bridge:intent`.** It hits the 1 MB `validateIntent` cap (Pitfall 3).
- **Storing the extracted document on the user message / in history.** Breaks D-05 freshness and bloats every subsequent turn (Pattern 3).
- **Editing `RESPONSE_CONTRACT_CORE`, `CARDINALITY_CHAT`, `CHAT_PERSONA`, `markerPreservationClauses`, or `scribeResponse.js`.** Frozen by D-07; verified byte-identical in v3.2-02 (`git diff 23eb4f1de`).
- **A literal "1. / 2. / 3." enumerated list in the prompt or any "tell the user which sources you used" instruction.** Invites Mistral-Small meta-commentary that breaks the JSON contract (GATE v3.1-03).

## Common Pitfalls

### Pitfall 1: `doc.GetAllParagraphs()` silently omits table-cell paragraphs
**What goes wrong:** Full-document markdown is missing all table content (or tables entirely), with no error.
**Why it happens:** `doc.GetAllParagraphs()` returns only document-level paragraphs; cell paragraphs are excluded. Verified by the codebase's own compensating loop + comment at `code.js:638` and the inject path's separate `doc.GetAllTables()` cell walk (`:639-653`). Note the selection path uses `range.GetAllParagraphs()` (a *range*), which DOES include cell paragraphs — hence its `insideTable` machinery. The two are not interchangeable.
**How to avoid:** Use the `doc.GetElement(i)` document-order walk (Pattern 1) which surfaces tables as first-class blocks; emit each via `extractTableCells`.
**Warning signs:** A test doc with a table produces markdown with no `[TABLE:…]` markers.

### Pitfall 2: `sendMessage` becomes truly async on the network-of-iframes
**What goes wrong:** The document block is empty because the code built `aiMessages` before the plugin replied.
**Why it happens:** `sendMessage` is already `async`, but today every input is synchronous; the extraction adds a real `await` on a postMessage round-trip across two iframe boundaries (Drive → OO editor → plugin).
**How to avoid:** `await extractFullDocument()` BEFORE assembling `aiMessages`; set `isLoading(true)` and add the user bubble FIRST (the existing "Scribe is thinking…" indicator covers the latency — DEC-UI-02). Always resolve the promise (timeout → `{ error: 'timeout' }`) so a missing/dead plugin yields the `'unavailable'` notice instead of a hung send.
**Warning signs:** Intermittent empty document context, especially on first send after panel open.

### Pitfall 3: the 1 MB `MAX_DATA_SIZE` cap (`protocol.js:35`)
**What goes wrong:** A large document's payload is rejected (`'data payload exceeds 1MB limit'`) and dropped — if routed through the intent protocol.
**Why it happens:** `validateIntent` (`protocol.js:99-146`) and `validateResponse` (`:155-201`) reject any `cozy-bridge:intent`/`:response` whose `JSON.stringify(data).length > 1_000_000`. `MAX_DATA_SIZE` is a *self-imposed validation*, NOT a `postMessage` transport limit (structured clone handles many MB).
**How to avoid:** Use a **dedicated message type** (Pattern 2) — it is not processed by `CozyBridge._listener`/`validateIntent` (which only handle `MSG_TYPE_INTENT`, `index.js:101`), so the cap does not apply. Rely on the budget (Pattern 5) to bound size; keep a sane hard ceiling (e.g. a few MB) above which you emit `'truncated'`/`'unavailable'` rather than chunk (chunking is unnecessary complexity here). **If** the planner instead chooses to reuse the intent protocol, the 1 MB cap becomes a hard constraint and bounding-before-send is mandatory.
**Warning signs:** `[cozy-bridge] Invalid intent: data payload exceeds 1MB limit` in console; document context works for small docs only.

### Pitfall 4: leaf helpers live inside the `callCommand` ES5 sandbox
**What goes wrong:** Reusing `paragraphToMarkdown` from a new function throws `ReferenceError`.
**Why it happens:** `callCommand` serializes its function and re-evaluates it in the sdkjs document context; closures over the plugin-iframe scope are lost. ALL leaf helpers are defined INSIDE the init `callCommand` body (`code.js:2072-2630`, ~555 lines).
**How to avoid:** Two options — (a) **Recommended:** factor the whole `callCommand` body (helpers + a `mode` branch reading `Asc.scope.scribeExtractMode = 'selection' | 'document'`) into ONE named function called by both init and the new on-demand handler (DRY; single fidelity source — directly serves D-01's "same rendering"). This DOES edit the selection path, so the **selection-case harness + `scribeResponse.corpus.spec.js` must stay green** and selection behavior must be byte-identical. (b) **Lower-risk:** a separate new `callCommand` that **duplicates** the needed leaf helpers (zero risk to the selection path, ~maintenance debt). State the choice explicitly in the plan.
**Warning signs:** Selection extraction regressions after refactor; or two diverging copies of `paragraphToMarkdown` drifting over time.
**ES5 reminder:** no arrow funcs, no `const`/`let` anywhere in `code.js` (project rule, MEMORY.md).

### Pitfall 5: redo-stack truncation from a read-write `callCommand`
**What goes wrong:** Sending with « document » checked breaks the user's redo.
**Why it happens:** A read-write `callCommand` truncates OO's redo stack; the init path guards this with `suppressExtractionUntil` after undo/redo (`code.js:2040,2057-2063`). The selection extraction uses read-write (`false, false`) to allow `SetName` on images (`code.js:2901-2902`).
**How to avoid:** Run the document extraction `callCommand` **read-only** — the document context is reference-only (never re-injected), so image `SetName` for stable `{{IMG:}}` markers is unnecessary. [Exact read-only flag = verify against the installed sdkjs `callCommand` signature — see Open Question 1; marked [ASSUMED] until confirmed.]
**Warning signs:** Redo unavailable after a chat send that included the document.

### Pitfall 6: determinism spec vs. fresh-each-send extraction (D-03 ↔ D-05)
**What goes wrong:** `ScribeContext.compose.spec.jsx`'s "byte-identical aiMessages across two sends" assertion fails once `includeDocument` is involved.
**Why it happens:** D-05 re-extracts each send; if the mock returns different md, the prompt differs.
**How to avoid:** In the spec, mock `extractFullDocument` to return a fixed md so determinism holds **for a fixed document state**. Document that the v3.2-02 determinism guarantee is now "same transcript + same checkboxes + same document state ⇒ same aiMessages" (the doc-state dependency is intentional per D-05).
**Warning signs:** Flaky/failed determinism assertions after wiring the document path.

### Pitfall 7: i18n parity + literal-audit gates
**What goes wrong:** CI fails on missing locale keys or hard-coded strings.
**Why it happens:** v3.1-07 gates enforce 5-locale parity (fr/en/de/es/it) and zero hard-coded user-facing strings; `scribeLiteralAudit.spec.js` is active.
**How to avoid:** Add `Scribe.include.documentTruncated` and `Scribe.include.documentUnavailable` (and optional `Scribe.loading.reading_document`) to ALL FIVE locale files with the strings in UI-SPEC; route every notice through `t()`.
**Warning signs:** `scribeLiteralAudit.spec.js` failures; missing-key warnings.

### Pitfall 8: footnote/cross-ref/image marker noise in read-only context
**What goes wrong:** The model sees `{{REF:scribe-ref-3:…}}`, `[^scribe-fn-2]`, `[TABLE:0]/[CELL:r,c]`, `{{IMG:…}}` markers in the document context and may echo them or get confused.
**Why it happens:** D-01 mandates verbatim reuse of the emitter, which produces these round-trip markers. Footnote *definitions* (the actual footnote text) are NOT emitted by the selection emitter (refs only — `code.js:2316-2326`; defs are handled separately at inject time via `GetFootnotesFirstParagraphs`, `:701`), so the document context will likewise carry footnote *references* without their text.
**How to avoid:** For v3.2-03, ship verbatim (faithful, reuses emitter, lowest risk) and do NOT add marker-preservation clauses for the doc block. Flag marker-simplification for the read-only context as a future refinement (Open Question 2).
**Warning signs:** Model output contains stray `{{REF:…}}`/`[^scribe-fn-…]` tokens, or asks about footnote contents it can't see.
