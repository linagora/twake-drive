# Phase v3.2-03 Plan 02: Host compose + budget + framing Summary

## Accomplishments
- `extractFullDocument()` in View.jsx: broadcasts `cozy-bridge:extract-document` with a fresh reqId, awaits the correlated `cozy-bridge:document-extracted` reply (matching type AND reqId), resolves `{ md, error }`; one-shot listener removed on success AND on an ~8s timeout (no leak, no hang). Injected into ScribeContext via `setExtractFullDocument` mirroring `setPanelActions`.
- `ScribeContext.sendMessage`: live-reads `includeDocumentRef`, awaits the extraction after the user bubble + `setIsLoading(true)` (so the existing "Scribe is thinking…" indicator covers latency), applies `applyBudget`, prepends an EPHEMERAL `[Full document]…[End of full document]` block to the CURRENT turn only (order document → selection → text), and records a `documentNotice` on the user message.
- `scribeAI.contextSourceFraming`: append-only `includeDocument` sentence + one combined doc+selection focus clause; `buildChatSystemPrompt` threads `includeDocument` (gated on actual inclusion).

## CTX-LLM-04 — Context size strategy (documented)

- **Config key:** `drive.scribe.contextBudget` (cozy-flags).
- **Unit:** characters (no client tokenizer; a chars≈tokens×4 heuristic is acceptable if a token unit is ever preferred).
- **Default:** unset (`null`/`undefined`) or `<= 0` ⇒ **UNLIMITED** — the whole document (already returned uncut by the plugin) is sent untouched, nothing is cut, no notice (D-02 default; never silent ⇒ silent success is correct).
- **Cut point:** when a positive budget is configured AND `md.length > budget`, keep the HEAD up to the budget and cut at the LAST `\n\n` block boundary at/under the budget (the emitter joins blocks with `\n\n`), so a partial trailing block is never sent. Head-keeping front-loads titles/headings/intro; the user's focus is covered separately by the selection block (D-04).
- **Sole truncation point:** `applyBudget` on the host. The plugin reply carries NO truncation field — there is no source-side truncation to react to.
- **Never-silent feedback:** truncation sets `userMessage.documentNotice = 'truncated'`; extraction error/timeout sets `'unavailable'` (no doc block, message still sent). Rendering of the notice is plan 03 (ChatMessageList + i18n).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Current-turn detection + prior-history snapshot hardened against id collisions and async-flush duplication**
- **Found during:** Task 2 (GREEN — the ephemeral/live-ref specs)
- **Issue:** The newly-introduced `await` between pushing the user bubble and assembling `aiMessages` exposed two latent defects. (a) Identifying the current turn by `m.id === userMessage.id` is unsound because `id: Date.now()` collides when two sends occur in the same millisecond (so prior turns wrongly received the document block). (b) Reading `messagesRef.current` AFTER the await could pick up a React-flushed, notice-updated copy of the current message and then append `userMessage` again, duplicating the current turn in history.
- **Fix:** Snapshot the prior history into a local `priorMessages` BEFORE the await; detect the current turn by object identity (`m === userMessage`) for both the doc-block prepend and the `documentNotice` map update.
- **Files modified:** src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx
- **Verification:** The "ephemeral (D-05)" and "live-ref toggle" specs pass; the existing v3.2-02 determinism/quadrant specs remain green (14/14 compose).
- **Committed in:** `a5699bdcc` (Task 2 GREEN commit)
