## Human Verification Required

### 1. Full-Document Markdown Fidelity (Tables + Headings)

**Test:** Open a real document in the OO dev environment with at least one table and several headings. Send a message with only « document » checked. Inspect the system prompt or network payload.
**Expected:** The `[Full document]...[End of full document]` block contains table cell content (`[TABLE:N]`, `[CELL:r,c]` markers) and heading prefixes (`## Title`), proving the `GetElement` walk (not `GetAllParagraphs`) reached table cells.
**Why human:** GetElement/GetClassType document-order walk correctness on real OO WASM documents requires the live OO container; Jest mocks cannot simulate this.

### 2. Redo Stack Preservation (Read-Only callCommand)

**Test:** Make several edits in the OO document, then send a message with « document » checked. Press Ctrl-Z multiple times.
**Expected:** All edits made before the send remain undoable; the document extraction did not consume or truncate the redo stack.
**Why human:** Correctness of `callCommand(fn, true /* read-only */)` against the installed sdkjs build requires the live OO container; the read-only contract is an OO runtime guarantee.

### 3. Truncation Notice End-to-End

**Test:** Set `drive.scribe.contextBudget` to a small value (e.g. 500) via cozy-flags/localStorage. Open a document with more than 500 characters of content. Check « document » and send a message.
**Expected:** A discreet info row appears directly below the user bubble in the chat thread with the text "Document too long — only part of it was sent to the assistant." (EN). The LLM still responds (message was sent with the truncated content).
**Why human:** End-to-end flow from extraction → applyBudget → documentNotice → ChatMessageList render requires the live Scribe panel.

### 4. Unavailability Notice (Timeout Path)

**Test:** Simulate the plugin being unresponsive (e.g. disable the plugin frame). Check « document » and send a message. Wait approximately 8 seconds.
**Expected:** The send completes (no hang); an ErrorBubble appears below the user bubble with the text "Couldn't read the document — your message was sent without it." The LLM receives and processes the message (without the document).
**Why human:** The 8s timeout fallback requires a real plugin absence to trigger; cannot simulate in Jest.
