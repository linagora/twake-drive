# Phase v3.2-03 Plan 01: Plugin Full-Document Extractor Summary

Added an on-demand whole-document markdown extraction path to the OnlyOffice plugin (`code.js`, ES5-strict) by factoring the existing per-element selection emitter into a reusable named function with a `selection|document` mode branch, then driving it over the entire document via a document-order `GetElement` walk and exposing it over a dedicated, production (un-gated) `reqId` request/response channel that bypasses the 1 MB cozy-bridge intent cap.

## What Was Built

### Task 1 — Reusable named extractor with a mode branch (commit c3e9e753a)
- Relocated the ~828-line inline `callCommand(function(){…})` body from inside `window.Asc.plugin.init` into a named IIFE-scope declaration `function buildScribeExtractionResult()`. Being a named declaration at plugin-iframe scope lets BOTH the init path and the new document-extract listener invoke the same fidelity source by name (D-01 "same rendering"). `callCommand` re-serializes the function body, so all leaf helpers remain defined INSIDE it (Pitfall 4 sandbox constraint).
- The init path now sets `window.Asc.scope.scribeExtractMode = "selection"` before calling `callCommand(buildScribeExtractionResult, false, false, <existing callback>)`. The read-write callback at the call site is untouched.
- Added a mode branch at the `// --- Main extraction logic ---` point: when `Asc.scope.scribeExtractMode === "document"` it returns `buildDocumentExtractionResult(doc)`; otherwise (`"selection"` / undefined) the existing selection logic runs UNCHANGED, returning the same JSON keys (`text`, `md`, `tableDocIndices`, `tableAmbiguity`, `partialTableInfo`, `crossRefMeta`).

### Task 2 — Document-order extraction + on-demand listener (commit 64a5ee7ab)
- Implemented `buildDocumentExtractionResult(doc)` inside the sandbox. It iterates top-level blocks via `doc.GetElementsCount()` / `doc.GetElement(i)` branching on `GetClassType()`: `"table"` → `[TABLE:N]…[/TABLE]` via `extractTableCells(el)`; `"paragraph"` → block-image marker (`getDrawingMarker`), heading/list prefixes (`getHeadingLevel` / `isListParagraph` + a document-level indent→depth map), and `paragraphToMarkdown(el, 0, 0)`. Blocks are joined with the same spacing rule as the selection path (`\n` between consecutive list items, `\n\n` otherwise). It deliberately does NOT use `doc.GetAllParagraphs()`, which silently omits table-cell paragraphs and would drop all table content (Pitfall 1).
- The document path applies NO plugin-side size guard and never returns a silent `md:""`: it always returns the WHOLE document (D-02). The selection path's silent `>100`-paragraph `md:""` fallback is NOT inherited. All bounding/never-silent truncation signalling is host-side (plan 02 `applyBudget`); the reply shape stays `{ type, reqId, md, error }` with no truncation field.
- Added an always-on `window.addEventListener("message", …)` listener (sibling of the selection-subscribe listener, NOT gated by any dev/test hook — it is a production feature) that filters `msg.type === "cozy-bridge:extract-document"`, seeds the marker counters, sets `scribeExtractMode = "document"`, runs the extractor **read-only** (`callCommand(buildScribeExtractionResult, true, false, cb)`), and replies via `postToAncestors({ type: "cozy-bridge:document-extracted", reqId, md, error })`. A dedicated message type bypasses `validateIntent`'s 1 MB cap (only `:intent`/`:response` are size-validated). On any thrown error or JSON parse failure it replies `md:""` with a non-null `error` string (the host maps `error` → 'unavailable').

## Read-only vs read-write decision (acceptance criterion)

The document `callCommand` is invoked **read-only** (`true`), per the RESEARCH Open Q1 resolution, to avoid truncating the user's redo stack (Pitfall 5). A scan of the shared extractor found exactly ONE document-mutating call — `drawing.SetName(name)` in `getDrawingMarker` (used to persist stable image marker names for selection re-injection). In read-only mode that write is a no-op or throws, so it is now guarded with `try { drawing.SetName(name); } catch (eSetName) {}`. The computed `name` is assigned before the write, so the marker is emitted into the markdown regardless. The selection path remains read-write, where `SetName` still succeeds and the guard never catches — selection behavior is byte-identical.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Guard the lone `SetName` write so the read-only document path cannot throw**
- **Found during:** Task 2
- **Issue:** The plan's read-only document `callCommand` shares `getDrawingMarker`, whose unconditional `drawing.SetName(name)` is a document write. Under a read-only `callCommand` this would throw on any document containing an unnamed image, failing the whole extraction for such documents (the most common real-world case).
- **Fix:** Wrapped the single `SetName` call in `try { … } catch (eSetName) {}`. Behaviorally byte-identical on the read-write selection path (SetName still succeeds there); makes the read-only document path robust (image markers still emitted with the computed name).
- **Files modified:** plugins/onlyoffice-scribe/scripts/code.js
- **Commit:** 64a5ee7ab
