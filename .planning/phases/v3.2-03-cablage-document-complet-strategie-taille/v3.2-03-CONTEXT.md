<domain>
## Phase Boundary

Troisième et **dernière** phase de v3.2 « Contexte enrichi du prompt ». Câble réellement la source de contexte **la plus coûteuse/nouvelle** : le **document docx complet**, extrait en **markdown**, gaté par la case « document » posée (état seul) en v3.2-01. C'est la seule des trois sources qui **n'a aucun chemin d'extraction existant** — le plugin OO n'extrait aujourd'hui que la **sélection**. Cette phase ajoute donc un **nouveau chemin d'extraction document-entier** côté plugin (réutilisant l'émetteur markdown par-élément déjà présent), définit la **stratégie de taille/troncature** (CTX-LLM-04) et évolue le **cadre de prompt énuméré** (graine D-05 posée en v3.2-02) pour informer le modèle des sources fournies.

Couvre **CTX-LLM-01** (injection du docx complet) et **CTX-LLM-04** (stratégie de taille + retour utilisateur si tronqué).

Le câblage se fait au **même point de composition unique** `ScribeContext.sendMessage` que v3.2-02, en réutilisant le seam déjà rendu extensible (booléens lus en live, `buildChatSystemPrompt` déjà paramétré par flags de source). Le **format conversationnel multi-tours** et le **contrat de réponse v3.1** (`discussion`/`fragments`) restent inchangés.

**Out of scope (NE PAS construire ici) :**
- Toute refonte du contrat de réponse v3.1 ou du re-ask/repli (figés).
- L'aplatissement du multi-tours en prompt unique.
- Un tokenizer client-side exact (on estime ; le budget réel vient de la config — voir D-02).
- Le câblage discussion/sélection (déjà livré en v3.2-02).
</domain>

<decisions>
## Implementation Decisions

### D-01 — Chemin d'extraction document-entier (nouveau, plugin)
- **Nouveau** chemin côté plugin OO (`code.js`) qui scanne **tout le document** via `Api.GetDocument().GetAllParagraphs()` (et tables/footnotes/refs/images) en **réutilisant l'émetteur markdown par-élément existant** (`paragraphToMarkdown` et co., aujourd'hui scopé à la sélection via `GetRangeBySelect`). **Pleine fidélité** : titres, listes, tables, footnotes, cross-refs `{{REF:…}}`, images — le même rendu que la sélection, mais sur le doc entier.
- **À la demande (request/response)**, PAS le push passif `SELECTION_CHANGED` : la sélection est poussée en live à chaque changement ; le document, lui, n'est extrait **que quand on en a besoin** (envoi avec « document » coché — voir D-05). ⇒ nouveau handler/intent dédié (nom + plomberie = discrétion ; ex. `EXTRACT_FULL_DOCUMENT` ou extension du payload de l'appel d'envoi).
- Plugin = **ES5 strict** (pas d'arrow/const/let).

### D-02 — Stratégie de taille = pilotée par config, **défaut illimité** (CTX-LLM-04)
- Le LLM utilisé **n'est pas connu et doit rester configurable** ⇒ coder un seuil de contexte en dur serait faux.
- La feature de **limitation est pilotée par configuration** : un **budget** (seuil de troncature) est *configurable*.
- **Par défaut, sans configuration : AUCUNE limite → on envoie le document entier.**
- Quand un budget **est** configuré et dépassé : on **tronque** (point de coupe documenté) et on **signale** la troncature (jamais silencieux — voir D-03).
- Le **chiffre exact, l'unité** (tokens vs caractères) et le **point de coupe** ne sont **pas figés en dur** : ils viennent de la config / du plan. Une **heuristique d'estimation** caractères→tokens est acceptable (pas de tokenizer client).
- Cela satisfait CTX-LLM-04 : la stratégie est **décidée et documentée** (« budget configurable, défaut illimité, troncature signalée si dépassée »).

### D-03 — Retour « tronqué » = jamais silencieux, indicateur discret (CTX-LLM-04, critère #3)
- Quand (et seulement quand) une troncature a lieu (cas budget-configuré-dépassé), l'utilisateur **en est informé explicitement** — **jamais de coupe silencieuse**.
- Forme : **indicateur discret + i18n** (style `SelectionChip` / zone « Inclure »), cohérent avec l'UX discrète de v3.2-01. La **forme exacte** (chip/badge dans la zone « Inclure », ligne système, tooltip ; avant vs après envoi) = **discrétion du planner**.
- ⚠️ Le **repli SILENCIEUX existant** du plugin (>100 paragraphes ⇒ texte brut `md:""`, `code.js` l.2638) NE doit PAS rester silencieux pour le chemin document : soit on le rend bavard, soit le nouveau chemin doc a son propre signalement.

### D-04 — Document + sélection cochés ensemble = les deux, indépendamment (critère #5)
- Si « document » ET « sélection » sont cochés : on envoie **le document complet ET** le bloc `[Selected text…]` de la sélection (la sélection est un sous-ensemble du doc, mais on ne dé-duplique pas).
- Le **cadre du prompt** (D-06) explique que la sélection est le **focus** de l'utilisateur dans le document. Comportement simple, déterministe, fidèle à « le modèle juge la pertinence ».
- Les **3 sources** (document / discussion / sélection) doivent être cochables dans **n'importe quelle combinaison** sans casser la composition ni le contrat (critère #5) — les fondations déterministes posées en v3.2-02 rendent ça direct.

### D-05 — Fraîcheur : extraction fraîche à chaque envoi
- À **chaque envoi** avec « document » coché, on **(re)demande** l'extraction au plugin. Toujours aligné sur l'état réel du docx, modèle mental simple, **zéro bug de cache périmé**.
- Coût accepté : un aller-retour plugin + latence sur gros docs à chaque tour. (Cache+invalidation explicitement **écarté** pour cette phase — risque de péremption si l'utilisateur édite entre deux tours.)

### D-06 — Composition + cadre énuméré (évolution de la graine D-05 de v3.2-02)
- Injection au **même point** `ScribeContext.sendMessage` (le seam), via un **nouveau bloc nommé « document »** dans le message courant / le prompt, **gaté par `includeDocument` lu en live** (mêmes refs/pattern que `includeSelection`/`includeDiscussion` posés en v3.2-02). `includeDocument` existe déjà (ScribeContext l.65-67, défaut décoché).
- On **fait grandir** la graine de framing D-05 (v3.2-02) en le **cadre énuméré** ciblé par l'utilisateur : « voici la demande ; tu peux t'appuyer sur 1. l'historique 2. la sélection 3. le document complet — juge ce qui est pertinent ». **Direction verrouillée** ; **formulation exacte = discrétion du planner** (anglais, comme le reste des prompts ; conditionnelle aux cases réellement cochées).
- HARD : le cadre **ne doit pas** affaiblir/contredire le contrat JSON (`RESPONSE_CONTRACT_CORE`), ni inciter au méta-commentaire — fragilité connue de Mistral-Small (cf. GATE v3.1-03). Re-validé par sonde + régression.

### D-07 — Garde-fou contrat v3.1 (NON négociable)
- Le contenu document injecté ne **fuit jamais** dans `fragments` ; séparation `discussion`/`fragments` préservée.
- `callScribeAIWithReask`, le repli contextuel, `scribeResponse.js` **intacts**.
- Sonde dev **PROBE-01** + corpus de régression v3.1 **au vert** avec le document activé (même gate que v3.2-02).

### Claude's Discretion (planner/researcher décide)
- Le **nom de l'intent** et la plomberie request/response de l'extraction document-entier (réutiliser le canal bridge existant vs nouveau ; cf. `protocol.js`).
- La **clé/forme de configuration** du budget (où elle vit, unité, valeur par défaut = illimité) et l'**heuristique d'estimation** de taille.
- L'**algorithme de point de coupe** quand on tronque (depuis le début / en réservant la place pour la réponse / autour de la sélection).
- La **forme UI exacte** de l'indicateur « tronqué » (D-03) et la **formulation** du cadre énuméré (D-06).
- La **limite plugin >100 paragraphes** (`code.js` l.2638) : la conserver/adapter/rendre bavarde pour le chemin doc.
- ⚠️ Le **cap transport 1 Mo** (`MAX_DATA_SIZE`, `protocol.js` l.35) s'applique au message bridge : sur un très gros doc, l'extraction peut le dépasser → à gérer (chunk, ou borne avant envoi) — c'est une contrainte technique réelle, pas une décision utilisateur.
</decisions>

## Existing Code Insights

### Reusable Assets
- **Émetteur markdown par-élément** (`code.js` l.2219+) : titres/listes/tables/footnotes/refs/images — déjà riche, **à réutiliser tel quel** pour le doc entier (le brancher sur `GetAllParagraphs()` au lieu de `GetRangeBySelect()`).
- **Seam de composition v3.2-02** : `buildChatSystemPrompt` paramétré par flags de source + booléens lus en live via refs — **conçu pour accueillir `includeDocument`** sans réécriture (D-00 γ de v3.2-02).
- **Canal bridge intent** (`castIntent`/`onIntent`) : réutilisable pour une requête document à la demande.
- **`SelectionChip` + zone « Inclure »** : patron UX discret pour l'indicateur « tronqué ».

### Established Patterns
- Multi-tours natif (`aiMessages`) — à conserver.
- Booléen d'inclusion lu **live** à l'envoi (refs), composition = `f(transcript, cases)` déterministe.
- Plugin OO = **ES5 strict** (pertinent ICI — le nouveau chemin d'extraction est dans `code.js`).

### Integration Points
- `includeDocument` (ScribeContext l.65-67) : mirrorer le pattern ref live et le lire dans `sendMessage`.
- Nouveau bloc « document » dans le message courant, indépendant de la sélection (D-04).
- Extraction = requête à l'envoi (D-05), pas le push `SELECTION_CHANGED`.
- Budget de taille = lecture d'une **config** (défaut illimité) avant troncature éventuelle + signalement.

### Critical Gaps (du scout)
1. **Aucune extraction document-entier** n'existe — nouveau handler + intent.
2. **Troncature aujourd'hui silencieuse** (>100 paragraphes) — à rendre explicite pour le doc.
3. **Cap transport 1 Mo** sur le message bridge — peut être dépassé par un très gros doc.
4. **Aucun budget en tokens** dans le code — tout est en octets.
</code_context>

<specifics>
## Specific Ideas
- Vision utilisateur (maintenant réalisée) : le cadre où le LLM est explicitement informé des sources fournies — « voici la demande ; tu peux t'appuyer sur 1. l'historique 2. la sélection 3. le document complet ; juge la pertinence ». v3.2-02 en a posé la graine ; v3.2-03 le construit.
- Principe de taille tranché par l'utilisateur : **« le LLM est inconnu/configurable → la limite est une config ; par défaut on envoie tout, et on ne tronque jamais en silence »**.
- Doc + sélection = les deux, indépendamment (pas de dé-duplication) ; le cadre dit que la sélection est le focus.
- Extraction fraîche à chaque envoi (pas de cache).
</specifics>
