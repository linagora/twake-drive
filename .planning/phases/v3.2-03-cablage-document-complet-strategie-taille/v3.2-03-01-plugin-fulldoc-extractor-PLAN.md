<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OO editor iframe → plugin iframe (postMessage) | Host broadcasts the extract request across nested iframes |
| plugin → host (postToAncestors) | Plugin returns potentially large document markdown |
| docx content → markdown payload | Untrusted document text is serialized for downstream LLM injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-v3.2-03-01 | Spoofing | new `cozy-bridge:extract-document` listener | mitigate | Filter strictly by `msg.type` and echo back the exact `msg.reqId`; same-page nested-iframe trust model identical to the existing selection-subscribe/dev-test channels; no secrets transit the channel |
| T-v3.2-03-02 | DoS | document extraction on huge docs (D-05 no cache) | mitigate | Reply is bounded downstream by the host budget (plan 02); the dedicated channel avoids the 1 MB intent path so it neither throws nor chunks; an in-plugin block-count guard surfaces `truncatedAtSource` instead of looping unbounded |
| T-v3.2-03-03 | Tampering | redo-stack truncation from a read-write callCommand | mitigate | Run the document callCommand read-only; fall back to the existing `suppressExtractionUntil` guard if read-only is unavailable |
| T-v3.2-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase (RESEARCH: zero new packages); no registry interaction |
</threat_model>
