<objective>
Make Replace and Insert buttons on panel AI messages actually modify the OnlyOffice document.

Purpose: The existing panelActions bridge calls `respond()` from useCozyBridge, but `respond()` silently no-ops when there's no pending intent (which is the case for pure chat flows where no AI_TEXT_ASSISTANT intent was emitted). From the user's perspective, the buttons do nothing.
Output: A new one-way `PANEL_ACTION` cozy-bridge intent type that carries `{action, text, html}` directly to the plugin, and a plugin-side handler that executes the same Replace/Insert pipeline as the existing response handler — without requiring a pending intent lookup.
</objective>

<problem>
**Bug confirmed in investigation** (2026-04-05):

`useCozyBridge.respond()` at `useCozyBridge.js:74-82` early-returns silently when `respondRef.current` is null:

```js
const respond = useCallback(responsePayload => {
  if (!respondRef.current) {
    console.warn('[useCozyBridge] No pending intent to respond to')
    return  // SILENT NO-OP
  }
  respondRef.current(responsePayload)
  respondRef.current = null
  setPendingIntent(null)
}, [])
```

`respondRef.current` is set only when the plugin sends an `AI_TEXT_ASSISTANT` intent (useCozyBridge.js:54). In the pure chat flow (user types a message in the panel, gets an AI reply, clicks Replace), no such intent was ever emitted, so respondRef is null and Replace/Insert buttons produce only a console warning.

Plugin-side (`code.js:1826-1827`) also requires a matching pending intent: `var pending = pendingIntents[msg.intentId]; if (!pending) return;` — so even if we hacked respond() to send something, the plugin would ignore it.
</problem>

<solution>
Introduce a new intent type `PANEL_ACTION` as a **one-way command** (not a response). It is distinct from the existing response mechanism:

- **Existing flow** (inline popover): plugin → `AI_TEXT_ASSISTANT` intent → Cozy Drive → user clicks → `respond()` response with matching intentId → plugin processes via pending intent handler.
- **New flow** (panel): user clicks → Cozy Drive → `PANEL_ACTION` intent → plugin processes directly without pending-intent lookup.

The plugin-side handler for `PANEL_ACTION` reuses the exact same code path as the response handler for replace/insert (same callCommand, same PasteHtml/Builder injection, same smart spacing). The only difference is the entry point.

**Benefits:**
- Clean separation of concerns: responses answer intents, actions are direct commands
- No hack on respond() — its semantics stay intact
- Plugin receives the same `{text, html, md, action}` payload shape in both paths, so the execution code is shared
</solution>
