<objective>
Replace volatile in-memory table clones (ApiTable.Copy()) with portable HTML snapshots captured at extraction time, so table formatting survives the full round-trip regardless of document changes.

Purpose: The current table pipeline relies on finding and cloning the original table in the document during injection. This is fragile: in the panel flow, the user may modify or delete the table between extraction and injection (multi-turn conversation, minutes of delay). Even in the inline flow, the clone only lives within a single callCommand scope. A portable HTML snapshot that travels with the enrichedMd makes the pipeline robust for both modes.

Output: At extraction, each table's complete HTML representation is captured via GetSelectedContent (or equivalent). At injection, the table is recreated via PasteHtml from this snapshot (preserving all formatting, images, merges, borders), then modified cells are updated with the LLM's content via the existing replaceCellContent() function.
</objective>

## Current Table Round-Trip (fragile)

**Extraction** (selection scan, inside callCommand):
- Tables detected via `doc.GetAllTables()` + range overlap checks (code.js:2514-2522)
- Cell content extracted as markdown via `extractTableCells()` / `extractPartialTableCells()` (code.js:2464-2487)
- Only TEXT captured — no formatting metadata (borders, bg, merges, font beyond first paragraph)
- Table document indices stored in `tableDocIndices[]` for later lookup
- Output: `enrichedMd` with `[TABLE:N][CELL:r,c]...[/CELL]...[/TABLE]` markers

**Injection** (inside callCommand for Replace/Insert):
- `tableClones = {}` created at line 779, populated at lines 873-998
- Original table found by document index: `doc.GetAllTables()[tableDocIndices[i]]`
- `ApiTable.Copy()` creates a volatile in-memory clone (lines 874, 963, 987)
- Cell content replaced in clone via `replaceCellContent()` (lines 831-855) using parsed LLM output
- Clone pushed to `content[]` array for `InsertContent` (line 1250)
- Font preserved per cell via `cFonts[r,c]` captured from original table (lines 918-947)

**Fragility points:**
1. `tableDocIndices` assumes table positions haven't changed — any insertion/deletion above shifts indices
2. `ApiTable.Copy()` clone lives only within the callCommand scope — not serializable, not storable
3. No formatting metadata travels with the enrichedMd — all formatting comes from the (possibly stale) original

<proposed_architecture>

## New Architecture: HTML Table Snapshots + PasteHtml Reconstruction

```
EXTRACTION (callCommand)                    TRANSPORT                        INJECTION
────────────────────────                    ─────────                        ─────────
For each TABLE:N in selection:              enrichedMd +                     For each [TABLE:N] in LLM response:
                                            tableHtmlSnapshots[]
│                                           travels via:                     │
├─ [TABLE:N][CELL:r,c] markers             - SELECTION_CHANGED              1. PasteHtml(tableHtmlSnapshots[N])
│  (text content → LLM)                    - AI_TEXT_ASSISTANT                 → recreates table with ALL original
│                                           - ScribeContext state               formatting (borders, bg, merges,
└─ tableHtmlSnapshots[N] =                 - PANEL_ACTION payload              font, images, everything)
     "<table>...</table>"                                                    │
     (HTML via GetSelectedContent                                           2. Get ApiTable ref to pasted table
     or equivalent OO API)                                                     (count tables before/after paste,
                                                                               or find by cursor position)
     Captures EVERYTHING:                                                    │
     - borders, colors, fonts                                               3. replaceCellContent() on cells
     - images (as base64/ref)                                                  that the LLM modified
     - merges, widths                                                          (existing code, unchanged)
     - any future OO properties                                              │
                                                                            4. Push modified table to content[]
                                                                               for InsertContent
```

**Key advantages over the previous structured-JSON approach:**
1. **Zero manual property extraction** — OO's HTML export captures everything automatically
2. **Images included** — base64 or internal refs, no separate pre-caching needed
3. **Future-proof** — new OO properties automatically captured in HTML, no maintenance needed
4. **Less code** — no captureTableSnapshot() with 15+ property readers, no rebuildTableFromSnapshot() with 15+ property setters
5. **Proven mechanism** — PasteHtml is already used in the existing pipeline for non-table content

**Key principle**: the HTML string is the snapshot. It is JSON-serializable (it's a string), self-contained, and captures everything OO renders. The only manual work at injection is modifying the cells that the LLM changed.

</proposed_architecture>

**ES5 syntax only. This is the critical task.**

Replace the current injection logic with the HTML snapshot approach:

**New injection flow for each [TABLE:N] in the LLM response:**

1. **Retrieve HTML snapshot**: get `tableHtmlSnapshots[N]` from the payload (received via Asc.scope or function parameter).

2. **PasteHtml the snapshot**: use executeMethod('PasteHtml', tableHtml) or the equivalent mechanism identified in Task 1 to insert the table from HTML. This recreates the table with ALL original formatting.

3. **Get ApiTable reference**: using the method identified in Task 1 (count before/after, cursor position, or other), get a reference to the just-pasted table.

4. **Modify cells with LLM content**: for each cell that the LLM modified (matched from parsed `[CELL:r,c]` output):
   - Get the cell via `table.GetRow(r).GetCell(c)`
   - Call the existing `replaceCellContent(cellContent, parsedCell, fontFamily, fontSize)` — use font from the pasted table's cell (already correct from HTML), not from a separate font cache
   - Cells NOT modified by the LLM are left untouched (they already have the correct content from PasteHtml)

5. **Push to content[] for InsertContent**: the modified table goes into the content array as before.

**What this replaces:**
- Remove `origTable = doc.GetAllTables()[tableDocIndices[i]]` lookups
- Remove `origTable.Copy()` volatile cloning (lines 874, 963, 987)
- Remove `cFonts` per-cell font capture from original (lines 918-947) — font comes from the pasted table cells directly
- Simplify partial table handling: PasteHtml the full table, modify only selected cells, done

**Timing considerations:**
- If PasteHtml is async (executeMethod), it cannot run inside callCommand. In that case, the injection sequence becomes:
  1. executeMethod PasteHtml for each table (sequenced via callbacks)
  2. callCommand to modify cells + build final content array
  This is a 2-step process. The Task 1 research should clarify whether this is needed.

**Backward compatibility:**
- If `tableHtmlSnapshots` is not present (e.g. older cached selection data), fall back to the existing Clone-from-document path. Check `if (tableHtmlSnapshots && tableHtmlSnapshots[N])` before using the new path.

**Error handling:**
- If PasteHtml fails or returns an unusable table, fall back to Clone path
- Log warnings for debugging: `log('[Scribe] Table HTML injection failed for TABLE:' + N + ', falling back to clone')`
