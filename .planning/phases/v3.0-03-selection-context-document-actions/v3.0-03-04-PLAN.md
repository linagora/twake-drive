<objective>
Replace the current markdown-to-HTML copy with a round-trip through the OnlyOffice engine to get OO-accurate HTML (tables, footnotes, cross-references, etc.).

Purpose: The current Copy implementation converts markdown → HTML via `marked.parse()` (MessageActions.jsx:55). This HTML is visually close but NOT identical to what OO produces — especially for complex constructs like tables, footnote references, and cross-references, which OO renders with specific structures that marked cannot replicate. We want max fidelity: the clipboard HTML should be exactly what the user would get if they had clicked Insert and then Ctrl+C'd the result.
Output: A new cozy-bridge request/response pair `REQUEST_RICH_HTML` where the plugin (a) inserts the markdown via the normal injection pipeline, (b) extracts the inserted content as HTML, (c) removes it, all within a single invisible operation, and returns the HTML. MessageActions awaits the response and writes it to the clipboard.
</objective>

<problem>
`MessageActions.jsx:53-69` currently does:
```js
const handleCopy = async () => {
  const html = stripParagraphWrapper(markdownToHtml(content).trim())
  const clipboardItem = new ClipboardItem({
    'text/html': new Blob([html], { type: 'text/html' }),
    'text/plain': new Blob([content], { type: 'text/plain' })
  })
  await navigator.clipboard.write([clipboardItem])
  // ...
}
```

The HTML comes from `markdownToHtml()` (marked.parse + remark-gfm), which produces generic HTML. OO renders tables, footnotes, and cross-references with its own DOM structure that marked cannot reproduce. For high-fidelity paste into OO or Word, users need OO-native HTML.

**What we want**: when the user clicks Copy, internally perform a "phantom insertion" through OO's engine, harvest the resulting HTML via OO's clipboard/export API, and remove the insertion — all invisibly.
</problem>

<approach>
**Fidelity-max approach with invisible round-trip:**

The plugin receives a `REQUEST_RICH_HTML` intent with the markdown content. It performs the round-trip inside OO using the **existing injection pipeline** (so the inserted content is guaranteed to match what Insert would produce), then extracts the HTML representation of the inserted content, removes it, and returns the HTML via a response.

**Invisibility strategies** (in order of preference, research during execution):

1. **Single callCommand with insert + extract + delete** — OO's callCommand is synchronous from the UI thread's perspective. If we can insert, call `GetSelectedContent` (or equivalent HTML export), and delete within the same callCommand, the user should not see any intermediate state because OO batches rendering.
   - Risk: callCommand creates ONE undo entry. Even if insert+delete net to zero, the undo history may show a phantom entry. Acceptable if it's a no-op that the user can simply skip over; unacceptable if it corrupts undo.
   - Mitigation: research whether OO supports `disableHistory` or a "transaction rollback" via the plugin API.

2. **Insert into an off-screen temporary location** — create a temporary paragraph far from the current scroll position (e.g. at document end), perform the insertion there, extract HTML, delete. The user never sees it because it's outside the viewport.
   - Risk: if the document is small or the viewport covers everything, it's still visible. Also, appending to document end and deleting may cause a scroll nudge.
   - Mitigation: use a bookmark region that's hidden or in a section that's not rendered.

3. **Document Builder temporary document** — spin up a separate Api document via the Builder API (if supported in plugin context), perform the build there, export HTML, discard the document.
   - Risk: the Document Builder API inside the plugin sandbox may not support creating standalone documents; it's typically scoped to the current document.
   - Mitigation: verify feasibility during task 1 research.

4. **Insert at cursor with screen refresh suppression** — use `executeMethod('PauseRendering')` or similar if it exists, do the round-trip, then `ResumeRendering`. Research whether OO exposes such an API.

**Task 1 must resolve which strategy is viable.** This is a research-heavy plan; the executor may need to spike multiple approaches.
</approach>
