## Pattern Assignments

### `ScribeResultPanel.jsx` (popover result card) — D-01/D-02/D-03

**Primary analog:** `FragmentCard.jsx` (the whole card to reuse) + its call site in `ChatMessageList.jsx`.

**The card to reuse verbatim** (`FragmentCard.jsx` lines 32-51) — accent-violet border, dark-aware bg, `MarkdownPreview` for cosmetic render, `MessageActions content={raw}` for RAW-fragment actions:
```jsx
const FragmentCard = ({ raw, hasSelection }) => {
  const theme = useTheme()
  const isDark = (theme.palette.type || theme.palette.mode) === 'dark'
  return (
    <div
      data-fragment-card
      style={{
        border: `1px solid ${SCRIBE_PURPLE}`,          // #7C3AED
        background: isDark ? 'rgba(124, 58, 237, 0.12)' : SCRIBE_PURPLE_08,
        borderRadius: 8,
        padding: '8px 10px',
        margin: '4px 0'
      }}
    >
      <MarkdownPreview>{raw}</MarkdownPreview>           {/* cosmetic only */}
      <MessageActions content={raw} hasSelection={hasSelection} />
    </div>
  )
}
```

**RAW-vs-cosmetic invariant (D-02)** — `FragmentCard.jsx` lines 18-30 spell out the governing rule the planner must preserve: `raw` is passed UNCHANGED to BOTH `MarkdownPreview` (which cleans markers internally, never mutating) AND `MessageActions` (which routes Copy/Insert/Replace on the verbatim raw). The popover must feed the card the **normalized RAW fragment**, not the display string.

**Where the popover already produces the right RAW value** (`ScribePopover.jsx` lines 240-261):
```jsx
// D-08 normalize parsed -> exactly one insertable fragment
let normalizedFragment
if (parsed.fragments.length >= 2) normalizedFragment = parsed.fragments.join('\n\n')
else if (parsed.fragments.length === 0) normalizedFragment = parsed.discussion
else normalizedFragment = parsed.fragments[0]
// ...
setRawResult(normalizedFragment)                 // <-- RAW (markers intact) for the card
setResult({ text: displayMd, breadcrumb, ... })  // displayMd is cosmetic-only today
```
> The card must render `rawResult` (markers intact), NOT `result.text` (`displayMd`).
> Today's `ScribeResultPanel` renders `resultText` (= `displayMd`) through `MarkdownPreview`
> at lines 1207. Aligning to the card means passing the RAW fragment in instead so
> `MarkdownPreview`'s own internal cleanup is the single cosmetic step (matching the chat).

**Current result render to replace/align** (`ScribeResultPanel.jsx` lines 1179-1211) — currently wraps `resultText` in a `scribe-result-text` div + `MarkdownPreview`, with a `cellWarning` banner. The card-aligned version reuses `FragmentCard`'s violet frame around `MarkdownPreview`. Keep the `cellWarning` banner (lines 1193-1206) above the card — it has no chat equivalent and is popover-specific.

**Action buttons divergence to preserve:** the popover keeps its OWN Insert/Replace/Retry footer (`ScribeResultPanel.jsx` lines 1279-1316, wired to `handleInsert`/`handleReplace`/`handleRetry` in `ScribePopover.jsx` l.345-361 which already operate on `rawResult`). Do NOT replace these with `FragmentCard`'s inline `MessageActions` icons — the popover footer is the established Insert/Replace surface and is already RAW-correct. Claude's Discretion (CONTEXT D-01 note): reuse `FragmentCard` for the *card frame + markdown*, or a thin popover wrapper, as long as the violet frame + RAW invariant hold. The card's own `MessageActions` may be redundant with the footer — planner decides whether to render the card body only (frame + `MarkdownPreview`) vs the full `FragmentCard`.

**Dev-mode branch is untouched:** `ScribeResultPanel.jsx` lines 1268-1277 (`devData ? <ScribeDevPanels> : resultContent`) — the card alignment applies ONLY to the `resultContent` (non-dev) branch.

**Discussion must NOT render (D-03 / INLINE-01):** the popover already feeds only `normalizedFragment` — never `parsed.discussion` as a visible bubble. The chat mirror (`ScribePopover.jsx` lines 270-291, `addMessage` with full `discussion`/`fragments`) is the INLINE-02 mirror — validate end-to-end, do not re-implement.

### `ScribeActionMenu.jsx` (integrated prompt entry + "open panel" entry) — D-04/D-05/D-06

**Analog:** the file's own existing prompt-input-at-index-`n` roving mechanics.

**Index `n` (prompt) anchor today** (`ScribeActionMenu.jsx` lines 77, 97-103, 386-390):
```jsx
const PROMPT_INDEX = actions.length                       // l.77 — prompt sits AFTER all actions
// updateFocus: l.97-103 — index===PROMPT_INDEX focuses the prompt input, else the menu Paper
// main-list arrows wrap over (actionCount + 1):                 // l.386-390
updateFocus((focusIndex + 1) % (actionCount + 1))
updateFocus((focusIndex - 1 + actionCount + 1) % (actionCount + 1))
```
> D-05 makes the prompt a *list entry* (today it's a separate wider `<Paper>` at lines 688-707,
> below the menu `<Paper>`). D-04 adds an "Ouvrir le panneau latéral" entry **after** the prompt.
> So the roving range grows from `actionCount + 1` (prompt) to `actionCount + 2`
> (prompt at `n`, open-panel at `n + 1`). Claude's Discretion (CONTEXT): planner confirms the
> exact index/roving; mirror the `% (count)` wrap pattern at lines 386-390.

**Prompt-input wiring to relocate into the list** (`ScribeActionMenu.jsx` lines 688-707) — the separate `<Paper width:500>` hosting `<ScribePromptInput>` with `onMouseEnter={updateFocus(PROMPT_INDEX)}` and highlight bg keyed on `focusIndex === PROMPT_INDEX`. `ScribePromptInput` itself (own file) already emits `onArrow('up'|'down')` / `onSubmit` / `onEscape` (lines 35-47) and `handlePromptArrow` (lines 136-163) already leaves the input to neighbours — extend `handlePromptArrow` so 'down' from the prompt goes to the new open-panel entry instead of wrapping to index 0.

**The open-panel callback to reuse for the new entry** (`ScribePopover.jsx` l.54, l.423): `onOpenPanel` is already threaded into `ScribeActionMenu`. D-06 removes the floating in-popover icon **button** (`ScribeActionMenu.jsx` lines 433-480, the `onOpenPanel && (<button><PanelIcon/>…</button>)` block + its tooltip) and replaces it with a normal `<ListItem>` at the bottom of the list that calls the same `onOpenPanel`. The `PanelIcon` svg (lines 31-36) can be reused as the `<ListItemIcon>`.

**ListItem entry pattern to copy for the open-panel entry** (`ScribeActionMenu.jsx` lines 579-600, the desktop main-list item) — `<ListItem button selected={…} onClick={…}><ListItemIcon><Icon …/></ListItemIcon><ListItemText primary={…}/></ListItem>` with first/last `borderRadius` corner handling at lines 583-584. The open-panel entry, being last, takes the `'0 0 8px 8px'` last-item radius (and the prompt entry, no longer last among real actions, loses it).

**i18n label (D-06):** reuse existing key `Scribe.button.open_panel` (already present in all locales, value `"Panneau Scribe"` in fr — `src/locales/fr.json:773`). The tooltip at `ScribeActionMenu.jsx` line 474 already uses `t('Scribe.button.open_panel')` — same key for the new `<ListItemText>`.

**Mobile parity:** the file has a full mobile submenu/roving branch (`isMobile`, lines 182-310, 488-562). The prompt already participates in mobile submenu nav via `PROMPT_SUBMENU_INDEX` (lines 180, 187-220). The planner must extend the mobile main-list roving (currently only the desktop `actions.map` at 565-684) to include the relocated prompt + open-panel entries consistently.

## Shared Patterns

### Scribe accent + dark-theme awareness
**Source:** `FragmentCard.jsx` lines 8-9, 33-34; `ChatMessageList.jsx` (SCRIBE_PURPLE / SCRIBE_PURPLE_08).
**Apply to:** the popover result card (must match chat exactly).
```js
const SCRIBE_PURPLE = '#7C3AED'
const SCRIBE_PURPLE_08 = 'rgba(124, 58, 237, 0.08)'
const isDark = (theme.palette.type || theme.palette.mode) === 'dark'
```
> The `(theme.palette.type || theme.palette.mode) === 'dark'` idiom is the
> project-wide dark check — used identically in `ScribeResultPanel.jsx` l.1177,
> `MarkdownPreview.jsx` l.23, `ScribeActionMenu.jsx` l.444. Reuse, don't reinvent.

### RAW-fragment action routing (the invariant)
**Source:** `FragmentCard.jsx` lines 18-48; `MessageActions.jsx` lines 79-91; `ScribePopover.jsx` lines 345-351.
**Apply to:** every Insert / Replace / Copy path in the popover result.
> Insert/Replace/Copy MUST receive the RAW normalized fragment (`rawResult`),
> never the cosmetic `displayMd`. `ScribePopover.handleInsert/handleReplace`
> already do `onInsert(rawResult || result.text)` — preserve this; the card render
> is display-only.

### MarkdownPreview as the single cosmetic step
**Source:** `MarkdownPreview.jsx` (whole component) + `tableCellMarkers.transformCellMarkersForPreview`.
**Apply to:** the popover card body (same component the chat card uses).
> `MarkdownPreview` internally runs the marker cleanup (REF/TABLE/footnote) — passing
> RAW in and getting clean display out, without mutating. This is why the card can take
> `raw` directly. The popover currently pre-transforms to `displayMd` before render
> (`ScribePopover.jsx` l.255-261); aligning to the card means letting `MarkdownPreview`
> do that cosmetic step on the RAW fragment, matching chat byte-for-byte.
