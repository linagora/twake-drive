<domain>
## Phase Boundary

Deuxième phase de v3.2 « Contexte enrichi du prompt ». Câble réellement dans le prompt LLM les **deux contextes bon-marché en mémoire** — l'historique de discussion et la sélection courante — **gatés par les cases « Inclure »** posées (état seul) en v3.2-01. Le câblage se fait au **point de composition unique** `ScribeContext.sendMessage`, en **réutilisant** l'encodage existant (`encodeSelectionForPrompt` + bloc `[Selected text from document]…[End of selected text]`, sérialisation d'historique `serializeAssistantTurnForHistory`). Le **format conversationnel multi-tours** et le **contrat de réponse v3.1** (`discussion`/`fragments`) restent inchangés.

Couvre CTX-LLM-02 (discussion), CTX-LLM-03 (sélection), CTX-LLM-05 (déterminisme + contrat préservé).

**Out of scope (NE PAS construire ici) :**
- Toute extraction document-entier (docx complet) → v3.2-03 (CTX-LLM-01).
- La stratégie de taille/troncature + feedback « tronqué » → v3.2-03 (CTX-LLM-04).
- Le **cadre de prompt énuméré explicite** complet (« 1. historique 2. sélection 3. document + juge la pertinence ») → conçu en v3.2-03 quand le document l'impose (voir Deferred). v3.2-02 ne fait qu'une **touche de framing additive minimale** (voir D-05).
- Tout abandon du multi-tours au profit d'un prompt unique reconstruit.
</domain>

<decisions>
## Implementation Decisions

### Architecture de prompt (le fork structurant — choix γ + touche β)
- **D-00 (direction) :** option **γ** — v3.2-02 **gate** la structure v3.1 existante (multi-tours + bloc `[Selected text…]` inline), et **pose/structure le point de composition unique** pour que v3.2-03 puisse y brancher le document et, si besoin, basculer vers un cadre nommé **sans réécriture**. Pas de redesign large maintenant (le contrat hard-gaté v3.1 + la sensibilité de Mistral-Small rendent un gros changement risqué).
- **D-05 (touche de framing additive) :** ajouter dans le **system prompt** une **courte annonce conditionnelle des sources de contexte réellement incluses ce tour-ci** (ex. « l'utilisateur peut fournir une sélection du document ; sers-toi de ce qui est pertinent, ignore le reste »). Conditionnelle aux cases, rédigée pour **ne pas perturber le contrat JSON** (pas d'incitation au méta-commentaire). Re-validée par sonde + régression. Le « juge la pertinence » reste **léger** — les cases font déjà le filtrage de pertinence côté utilisateur.

### Gating « discussion » (CTX-LLM-02)
- **D-01 :** « discussion » **décochée ⇒ tour courant seul** : system prompt + message courant (+ sélection si « sélection » cochée), **aucun tour antérieur**. Cochée ⇒ historique complet (comportement v3.1).
- **D-01a (changement de comportement assumé) :** v3.1 envoyait **toujours** l'historique complet ; `includeDiscussion` ayant pour défaut **décoché** (v3.2-01), le comportement par défaut devient **« pas d'historique au-delà du tour courant »**. C'est voulu et conforme au critère de succès 2 (historique = opt-in explicite).

### Gating « sélection » (CTX-LLM-03)
- **D-02 :** « sélection » **ne gate que le tour COURANT** (le chip). Les sélections des **tours passés** font partie de l'historique : elles sont incluses **ssi « discussion » est cochée**, et **rejouées telles quelles** (`[Selected text…]` inline préservé) → historique intelligible. Décocher « sélection » ne touche pas le passé.
- **D-04 (portée du décochage sur le system prompt) :** « sélection » décochée ⇒ **tout retiré pour le tour courant** : pas de bloc `[Selected text…]` ET `buildChatSystemPrompt` appelé **sans `selectionMd`** (donc pas de marker-preservation clauses). Le LLM ne voit **rien** de la sélection courante. Cohérent : décoché = la sélection n'influence pas le prompt. (Implémentation pressentie : gater à l'envoi en n'attachant pas `.selection` au message + ne pas passer `selectionMd` au system prompt.)

### Modèle d'état : « on ne stocke que ce qui est envoyé »
- **D-03 :** **aucun flag de coche persisté par message.** La composition = fonction pure `f(transcript, valeur courante des cases)`, cases lues **en live à l'envoi**.
  - **Sélection** : la donnée envoyée vit **dans le message** (`m.selection` attaché au moment de l'envoi ssi « sélection » cochée). Le transcript EST le registre fidèle de ce qui est parti ; rejeu = ressortir tel quel. Déjà le comportement actuel.
  - **Discussion** : **rien à stocker** — on ne reconstruit jamais un appel passé (chaque envoi est un appel réseau distinct, déjà parti/répondu) ; la case décide seulement ce que **l'appel courant** inclut du transcript (toujours présent pour l'affichage).
  - Conséquence : **même transcript + mêmes cases ⇒ même prompt** → déterminisme (CTX-LLM-05, critère 3). L'option « snapshot des cases par message » est explicitement **rejetée** (inutile, et pour « discussion » contre-intuitive — cocher après coup ne ferait pas remonter les tours créés case décochée).

### Garde-fou contrat v3.1 (CTX-LLM-05, critères 4-5) — NON négociable
- Les contextes injectés ne **fuient jamais** dans `fragments` ; séparation `discussion`/`fragments` préservée.
- Re-ask correctif unique (`callScribeAIWithReask`) + repli contextuel **intacts**.
- Sonde dev **PROBE-01** + corpus de régression v3.1 **au vert** avec les contextes activés.

### Claude's Discretion (le planner/researcher décide)
- Le **point exact** où poser le gate sélection (dans `ChatInput` en n'appelant pas `sendMessage` avec `selectionContext`, vs dans `sendMessage`) — D-04 fixe le comportement observable, pas la plomberie.
- La **formulation exacte** de la ligne de framing D-05 (wording anglais, comme le reste des prompts) et sa **conditionnalité** précise par case.
- La forme du **« seam » de composition** pour qu'il soit extensible au document v3.2-03 (sans sur-anticiper le cadre énuméré).
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `encodeSelectionForPrompt(sel)` → `{ selectionMd, markerClauses }` : source unique pour encoder la sélection (réutiliser, ne pas réécrire).
- `buildChatSystemPrompt(selectionMd)` : system prompt composable ; passer `undefined`/`''` quand sélection non incluse (D-04) supprime naturellement les marker clauses.
- `serializeAssistantTurnForHistory(parsed)` : sérialisation discussion-seulement des tours assistant (déjà discussion-only, ne dump pas les fragments bruts).

### Established Patterns
- Multi-tours natif (tableau `aiMessages` à rôles) — **à conserver** (D-00).
- Sélection inline dans le message user via bloc `[Selected text from document]…[End of selected text]` — format **gaté hard en v3.1, à ne pas modifier**.
- Plugin OO = ES5 strict (non concerné ici — câblage purement côté `src/` React).

### Integration Points
- Le gate « discussion » s'applique au **filtre/map de `currentMessages`** dans `sendMessage` (inclure ou non les tours antérieurs au courant).
- Le gate « sélection » s'applique (a) à l'attachement de `.selection` au tour courant et (b) à l'argument `selectionMd` de `buildChatSystemPrompt` (D-04).
- Les booléens viennent de `ScribeContext` (déjà dans le provider) — lecture live dans `sendMessage`.
</code_context>

<specifics>
## Specific Ideas

- Vision utilisateur (cible v3.2-03) : un cadre où le LLM est explicitement informé des sources fournies — « voici la demande ; tu peux t'appuyer sur : 1. l'historique, 2. la sélection, 3. le document complet ; juge ce qui est pertinent ». v3.2-02 en pose seulement la **graine** (D-05 framing léger) sans construire le cadre énuméré complet.
- Principe directeur retenu de la discussion : **« on n'envoie que ce qui est coché, et le transcript est le registre fidèle de ce qui a été envoyé »** (D-03) — pas d'état de coche à mémoriser.
- Avis technique acté : multi-tours pour la conversation + blocs nommés pour les ressources ; ne pas aplatir en prompt unique ; framing léger pour éviter de casser le contrat JSON sur un petit modèle (Mistral-Small).
</specifics>
