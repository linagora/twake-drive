## Summary

Reviewed the diff against base `23eb4f1de` for the v3.2-02 wiring that gates discussion + selection
into the chat prompt at the `ScribeContext.sendMessage` seam, plus the conditional D-05
`contextSourceFraming` line in `buildChatSystemPrompt`.

The core gate logic is correct and well-tested. I traced all four sel×discussion quadrants, the D-02
past-selection replay, and the D-03 live-read-via-refs path; the production prompt assembly behaves as
the plan specifies. The 35 new/updated tests pass. The new framing is static English with no user-data
interpolation, so it introduces **no new prompt-injection surface** and does not weaken the contract
strings (verified by the byte-identical regression assertion).

The two findings worth fixing are a **dev/probe data divergence**: when « sélection » is OFF the prompt
correctly sends no selection markdown, but the PROBE-01 sample and the dev-panel `_devExchange.devData`
still read the raw (excluded) `selectionContext`. Because the plan stores prod-gating decisions away
from the probe input, the dev conformance corpus that gates future prompt changes (per the v3.1-03 GATE
harness) gets fed input the model never saw, which can produce false REF/table "drop" flags. Production
message behavior is unaffected.

## Warnings

### WR-01: PROBE-01 sample records selection markdown that was NOT sent to the model when « sélection » is OFF

**File:** `src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx:271-278`
**Issue:**
The production prompt correctly gates the selection: `currentSelectionMd = selectionIncluded ? encodedSelectionMd : ''`
(line 195) and the current user turn carries no `[Selected text…]` block when `selectionIncluded` is false.
But the probe sample below it is NOT gated on `selectionIncluded`:

```js
recordProbeSample(parsed, {
  surface: 'chat',
  inputMd: selectionContext
    ? selectionContext.markdown || selectionContext.text || ''
    : '',
  ts: Date.now()
})
```

`recordProbeSample` derives `hasTable`/`hasRef` tags from `inputMd` (`deriveContentTags`) and computes
`refIntegrity(inputMd, fragments)` / dup metrics against it (`scribeProbe.js:374-383, 449`). So in
Quadrant B/D (selection OFF) the probe is told the model received a TABLE/REF selection it never saw,
and will flag the response for "dropping" markers that were correctly absent from the prompt. The inline
comment claims this is "the same `markdown || text` accessor used in the history mapping above" — but the
history mapping is now gated by `selectionIncluded`, so the two have drifted apart. This corrupts the
dev-mode conformance corpus that gates future prompt iterations (memory: v3.1-03 GATE harness relies on
probe accuracy). Dev-only (guarded by `isScribeDevMd()`), so no production impact — but it silently
poisons the measurement the project depends on.

**Fix:** Gate the probe `inputMd` on the same `selectionIncluded` decision (and reuse the already-encoded
md so the probe sees exactly what the prompt sent):

```js
if (isScribeDevMd()) {
  recordProbeSample(parsed, {
    surface: 'chat',
    inputMd: currentSelectionMd, // '' when selection excluded — matches what was sent
    ts: Date.now()
  })
}
```

### WR-02: Dev-panel `_devExchange.devData` mixes gated and ungated selection fields (inconsistent dev view when « sélection » is OFF)

**File:** `src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx:287-300`
**Issue:**
Within the same `isScribeDevMd()` branch, `md` is correctly gated (`md: currentSelectionMd` → `''` when
excluded) but `html`, `enrichedMd`, and `source` read the raw `selectionContext` unconditionally:

```js
devData: {
  html: selectionContext?.html || '',
  normalizedHtml: '',
  enrichedMd: selectionContext?.markdown || '',
  md: currentSelectionMd,
  source: selectionContext?.markdown ? 'plugin' : 'turndown'
}
```

When « sélection » is OFF the dev panel will show a non-empty `enrichedMd`/`html` and `source: 'plugin'`
alongside an empty `md`, implying a selection was part of the prompt when it was not. This makes the dev
surface misleading for exactly the case the phase introduces (selection excluded). Same root cause as
WR-01: the gating decision is not threaded through to the dev/observability fields. Dev-only, no
production impact.

**Fix:** Gate the selection-derived dev fields on `selectionIncluded` so the dev view reflects what was
actually sent:

```js
devData: {
  html: selectionIncluded ? selectionContext?.html || '' : '',
  normalizedHtml: '',
  enrichedMd: selectionIncluded ? selectionContext?.markdown || '' : '',
  md: currentSelectionMd,
  source: selectionIncluded && selectionContext?.markdown ? 'plugin' : 'turndown'
}
```

## Info

### IN-01: Quadrant D emits a `[Selected text…]` block in history with no corresponding selection framing in the system prompt

**File:** `src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx:200-209`
**Issue:**
When « sélection » is OFF but « discussion » is ON (Quadrant D), `contextSourceFraming` omits the
"The user may include a selection from the document…" line (because `includeSelection` is false), yet a
replayed prior user turn still contains its stored `[Selected text from document]` block (D-02 replay).
The model therefore sees a `[Selected text…]` block with no framing announcing that selections can
appear. This is the **documented, intended** D-02 behavior per the plan (selection framing describes only
the current turn; past selections replay unchanged), and the contract markers themselves are
self-describing — so this is not a defect. Noted only so a future reader does not mistake the asymmetry
for a bug. No change required.

**Fix:** None required. If desired in v3.2-03, the enumerated multi-source frame can make the
history-may-contain-selections case explicit.

### IN-02: `id: Date.now()` / `Date.now() + 1` collision risk on rapid sends (pre-existing, surfaced by review)

**File:** `src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx:172, 307, 322`
**Issue:**
User-message id is `Date.now()` and assistant/error ids are `Date.now() + 1`. Two sends within the same
millisecond, or a turn whose round-trip completes inside 1ms (the mocked test path), can collide and
produce duplicate React keys / ambiguous message identity. This predates v3.2-02 (the diff did not touch
the id scheme) and is out of strict scope, but the phase adds a deterministic test harness that drives
sends back-to-back, making the collision more reachable. No correctness impact on prompt assembly.

**Fix:** Use a monotonic counter or `crypto.randomUUID()` for message ids instead of timestamp
arithmetic. Out of scope for this phase; track separately.
