# Phase 27: Diagnostic Results — Footnotes & Cross-References

**Tested:** 2026-04-02
**OO Version:** 9.3.0-138
**Confidence:** HIGH (empirically verified on running container)

## Detection Strategy — CONFIRMED

### Cross-References → `type=hyperlink` with `GetLinkedText() === ""`

Cross-references (renvois) in OO are exposed as `hyperlink` elements in paragraph iteration. They are distinguished from regular external hyperlinks by having an **empty `GetLinkedText()`** (regular hyperlinks have a URL there).

| Cross-ref type | GetLinkedText | GetDisplayedText | GetScreenTipText | How to identify target |
|---|---|---|---|---|
| Heading ref | `""` | visible text (e.g. "vers un titre") | `"#_Heading_Text"` (starts with `#_`, underscored) | Strip `#_` prefix, replace `_` with spaces → heading text |
| Bookmark ref | `""` | visible text (e.g. "vers un signet") | bookmark name (e.g. `"un_signet"`) | Match against `doc.GetAllBookmarksNames()` |
| Footnote ref | `""` | visible text | TBD (not tested separately) | Likely similar pattern |

**Sub-elements:** Each hyperlink contains 1 sub-element (a run with the displayed text).

**hasCopy:** `false` — **Copy() is NOT available on hyperlinks**. Must use API recreation.

**Available methods on hyperlink:** `SetLink`, `SetDisplayedText`, `SetScreenTipText`, `GetLinkedText`, `GetDisplayedText`, `GetScreenTipText`, `GetElement`, `GetElementsCount`, `SetDefaultStyle`, `GetRange`, `ToJSON`

### Footnote Calls → `type=run` with `GetStyle().GetName() === "footnote reference"`

Footnote call markers (the superscript number in text body) appear as regular `run` elements with:
- `GetText()` → empty string (the number is auto-generated by OO)
- `GetStyle()` → returns a style object
- `GetStyle().GetName()` → `"footnote reference"`
- `GetStyle().GetType()` → `"run"`

Same pattern appears in the footnote content paragraph (PARA 8 elem 0) — the auto-number at the start of the footnote text.

**hasCopy:** `true` — Copy() IS available on these runs.

### Unsupported Elements → Opaque, Ignore

Elements with `type=unsupported` return `"unsupported"` from `ff()` and have only 2 props: `{ff, GetClassType}`. These are internal field markers (bookmark start/end, field begin/separate/end). They cannot be read, copied, or manipulated. **Ignore them during extraction.**

They appear in:
- Heading paragraphs (4 unsupported elements — likely TOC bookmark markers)
- Bookmark paragraphs (3 unsupported elements — bookmark start/end markers)

## Implementation Implications

### Extraction (paragraphToMarkdown)
1. **Cross-refs:** In the existing `hyperlink` branch, check `GetLinkedText() === ""` → it's a cross-ref, not a regular link. Extract `GetScreenTipText()` as target identifier, `GetDisplayedText()` as visible text. Emit `{{REF:scribe-ref-N:visible text}}`.
2. **Footnote calls:** In the `run` branch, check `el.GetStyle()?.GetName() === "footnote reference"` → emit `[^scribe-fn-N]`.

### Injection (addRunsToParagraph)
1. **Cross-refs:** Cannot Copy() hyperlinks. Must recreate via `AddHeadingCrossRef()` / `AddBookmarkCrossRef()` using cached metadata (target type + screen tip text).
2. **Footnote calls:** Can Copy() the run. Use same cache pattern as images. BUT: the run has empty text — need to verify that Copy() + AddElement preserves the footnote link.

### Metadata to cache during extraction
```
crossRefMeta: {
  "scribe-ref-0": { type: "heading", screenTip: "#_Un_titre", displayedText: "vers un titre" },
  "scribe-ref-1": { type: "bookmark", screenTip: "un_signet", displayedText: "vers un signet" }
}
```

Target identification during injection:
- Heading: strip `#_` from screenTip, replace `_` with space → find in `GetAllHeadingParagraphs()` by text match
- Bookmark: screenTip IS the bookmark name → pass directly to `AddBookmarkCrossRef()`
