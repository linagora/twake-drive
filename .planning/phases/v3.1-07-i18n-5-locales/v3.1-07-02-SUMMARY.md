**The 3 confirmed user-facing FR literals in ScribeResultPanel are wired through t() (panel.description, button.copy, result.table_partial_insert_unavailable), and a permanent source-scanning Jest gate (scribeLiteralAudit.spec.js) now fails if any known hard-coded user-facing literal reappears in the popover/chat/result surfaces.**

## Performance

- **Duration:** ~12 min
- **Tasks:** 2
- **Files modified:** 2 (1 created, 1 modified)

## Accomplishments

- Extracted the disabled-Insert tooltip, the result-panel info button (aria-label + title), and the copy affordance (aria-label + title) to t() calls, reusing the keys plan 01 added to all 5 locales.
- Added `const { t } = useI18n()` inside `DevPanelTitle` and `CopyButton` (the user-facing info/copy affordances) which previously had no i18n hook; the `ScribeResultPanel` component already had `t` in scope (no duplicate added).
- Created `scribeLiteralAudit.spec.js` — a dependency-free source-scanning regression gate over the 6 user-facing surfaces, proven to fail on re-introduction of a hard-coded prop literal and green against the extracted sources (with the out-of-scope dev-panel French prose still present in the file).
