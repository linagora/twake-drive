<domain>
## Phase Boundary

Last phase of milestone v3.1. Delivers **I18N-01**: the Scribe card labels and fallback/error messages are translated and verified across the **5 target locales — fr, en, de, es, it** — and **no hard-coded Scribe UI string remains** in the popover/chat/result surfaces.

This phase reuses the existing i18n infrastructure (cozy-ui `useI18n`/`t()`, `src/locales/{locale}.json` with a `Scribe.*` namespace). It does NOT introduce new locales (16 locale files exist; only the 5 above are in scope for verification), does NOT touch the floating `PanelIcon` document button, and does NOT change Scribe behavior — it is a translation-completeness + literal-audit phase only.

**Out of scope:** anything outside i18n (re-ask, corpus, response_format → done in v3.1-06; popover/chat rendering → v3.1-04/05). Non-target locales (ar, ja, ko, nl, pl, ru, vi, zh_*, nl_NL) may be left as-is — fixing them is optional, not a success criterion.
</domain>

<decisions>
## Implementation Decisions

### Locale source of truth (locked — codebase fact)
- `fr.json` and `en.json` are the **reference** locales: identical `Scribe.*` key sets, **44 keys each**, full parity (verified — zero diff between fr and en key sets).
- `de.json`, `es.json`, `it.json` each have **37 `Scribe.*` keys → 7 missing**, identical gap across all three:
  `panel.tagline`, `chat.typing`, `chat.error_prefix`, `chat.welcome`, `chat.selection_context`, `button.open_panel`, `button.copy`.
- The 7 missing keys must be added to de/es/it with proper translations (not English copies), matching the existing tone of each locale's other Scribe entries.

### Hard-coded string audit (locked — success criterion 2)
Seed findings from recon (the planner MUST run a fuller literal audit across all Scribe surface components, not just these):
- `ScribeResultPanel.jsx:215-216` — `"Description du panneau"` / `"Description"` (hard-coded FR aria/label).
- `ScribeResultPanel.jsx:269-270` — `"Copier le contenu"` / `"Copier"` (hard-coded FR — note `Scribe.button.copy` already exists and should be reused).
- `ScribeResultPanel.jsx:1331` — `"Insertion non disponible pour une selection partielle de tableau"` (hard-coded FR user-facing message).
- Each hard-coded UI string → extract to a new `Scribe.*` key, wire through `t()`, and add the key+translation to all 5 target locales.
- **Not** in scope as "strings": code constants like `'AbortError'`, `'ArrowLeft'`, `'ArrowRight'`, DOM/event names, className/variant/color/test-id props.

### Surfaces to audit (locked)
The popover/chat/result Scribe surfaces, at minimum:
`FragmentCard.jsx`, `ScribePopover.jsx`, `ScribeContext.jsx`, `ScribeResultPanel.jsx`, `ScribeActionMenu.jsx`, `ChatMessageList*.jsx`, `MarkdownPreview.jsx`, and the surface-facing strings produced by `scribeResponse.js` fallback path (`buildFallback` warnings are codes, not UI text — confirm they are not rendered raw to the user).

### Claude's Discretion (planner decides)
- Whether to split into two plans (Plan A: add 7 missing keys to de/es/it; Plan B: literal audit + extract hard-coded strings + add new keys to all 5) or keep as one — the two workstreams touch mostly disjoint surfaces and could parallelize.
- Exact translation wording for de/es/it (use existing locale tone; native-quality, not literal English).
- The shape of the automated verification (a Jest test asserting key-set parity across fr/en/de/es/it for the `Scribe.*` namespace is strongly recommended as the regression net for criterion 1; a lint/grep gate for criterion 2).
</decisions>
