  <action>
Modify `.scribe-result-panel` in scribe.styl:
- Remove `width 380px` (fixed width)
- Add `min-width 300px` to prevent the panel from collapsing too narrow
- Add `max-width 560px` to cap width (wider than menu's 220px min, narrower than prompt's 500px -- leaves room)
- Add `width fit-content` so the panel width adapts to text content
- Keep `padding 16px`

Modify `.scribe-result-text`:
- Change `max-height 300px` to `max-height min(400px, 60vh)` to use viewport-relative capping (per decision: ~400px or 60vh)
- Add `min-height 48px` to prevent an awkwardly short panel for very short results
- Add `transition max-height 200ms ease-out` for smooth resize animation when content changes
- Keep `overflow-y auto`, `padding 12px`, `border-radius 4px`, `font-family monospace`, `font-size 13px`, `line-height 1.5`, `white-space pre-wrap`, `margin-bottom 12px`

Note: The `transition` on `max-height` won't animate actual content-driven height changes (since max-height is constant). Instead, the real animation is the content appearing/changing. The CSS transition will smooth any programmatic max-height changes. For content-driven growth, the natural DOM reflow handles it. If the panel should animate height smoothly on content change, we need a different approach -- but per Claude's discretion, CSS transition on max-height is the simplest acceptable solution. The visual effect is: panel has a max boundary that it grows toward naturally.

Do NOT add any animation-related properties beyond the single transition on max-height. The deferred decisions explicitly exclude animations as a separate phase concern.
