## Warnings

### WR-01: ArrowUp on oldest card fires `e.preventDefault()` then returns — blocks container scroll

**File:** `src/modules/views/OnlyOffice/Scribe/ChatMessageList.jsx:216,245`

**Issue:** `e.preventDefault()` is unconditional for all arrow keys once `pos` is
found (line 216). The early-return guard for the oldest card comes *after* that
call (line 245). When focus is on any button of `groups[0]` and the user presses
ArrowUp, the browser's default action (scrolling the `overflowY: auto` container
upward) is prevented, and focus does not move either. The user is stuck: ArrowUp
does nothing and also prevents scrolling. All other card positions work correctly.

**Fix:** Move the `ArrowUp` boundary return *before* `e.preventDefault()`, or
conditionally prevent default only when the navigation action will actually execute:

```js
// ArrowUp
if (key === 'ArrowUp') {
  if (cardPos === 0) return // allow native scroll — no preventDefault
  e.preventDefault()
  const prevGroup = groups[cardPos - 1]
  const clampedUp = Math.min(buttonPos, prevGroup.length - 1)
  prevGroup[clampedUp].focus()
  return
}
```

Alternatively restructure to call `e.preventDefault()` only inside each branch
that actually moves focus, rather than centrally.

### WR-02: `MessageActions` — `setTimeout` confirmation-reset has no unmount cleanup

**File:** `src/modules/views/OnlyOffice/Scribe/MessageActions.jsx:58`

**Issue:** `showConfirmation` sets `timeoutRef.current = setTimeout(...)` but there
is no `useEffect` return that calls `clearTimeout(timeoutRef.current)` on unmount.
If a card unmounts while the 1500 ms timer is in flight (e.g., the panel closes
immediately after a Copy action), the `setConfirmedAction(null)` callback fires on
an unmounted component. React 18 silently drops the state update, so no crash
occurs, but the timer is leaked for up to 1.5 s on every panel close that follows
an action.

**Fix:**
```js
useEffect(() => {
  return () => {
    if (timeoutRef.current) clearTimeout(timeoutRef.current)
  }
}, [])
```

### WR-03: `MarkdownPreview` `a` renderer spreads model-controlled attributes without `rel="noopener noreferrer"`

**File:** `src/modules/views/OnlyOffice/Scribe/MarkdownPreview.jsx:63-65`

**Issue:** The custom `a` component does:

```jsx
a: ({ node, children: aChildren, ...props }) => (
  <a style={{ color: theme.palette.primary.main }} {...props}>
```

`...props` includes `href`, `title`, and crucially `target` if the model produces
raw HTML (via `rehype-raw`) such as `<a href="x" target="_blank">`. Without
`rel="noopener noreferrer"`, a model-controlled link opened in a new tab can set
`window.opener.location` (reverse tabnapping). React 18 sanitises `javascript:`
and `data:` URIs in `href`, so script-execution via `href` is mitigated, but the
opener relationship is not.

**Fix:**
```jsx
a: ({ node, children: aChildren, ...props }) => (
  <a
    style={{ color: theme.palette.primary.main }}
    rel="noopener noreferrer"
    {...props}
  >
    {aChildren}
  </a>
),
```

Placing `rel` before `{...props}` ensures it is set as the default; `{...props}`
can only override it if the hast node itself carries a `rel` attribute (which would
come from a model-authored link — acceptable, since the model's explicit `rel` is
at least as controlled as ours).

### WR-04: `rehype-raw` renders model-controlled HTML embedded in REF text and CELL content (acknowledged deferral — residual risk note)

**File:** `src/modules/views/OnlyOffice/Scribe/MarkdownPreview.jsx:153,163`

**Issue:** Two preprocessing steps inject model-controlled substrings into the
markdown source that is subsequently rendered by `react-markdown` with `rehype-raw`
active:

1. **REF text** — `out.replace(/\{\{REF:scribe-ref-\d+:([^}]*)\}\}/g, '$1')`:
   the capture group `[^}]*` allows any character except `}`, including `<` and `>`.
   If a model response contains `{{REF:scribe-ref-1:<b>bold</b>}}`, the replacement
   inserts raw `<b>bold</b>` inline in the markdown. remark parses this as an
   inline HTML node; rehype-raw passes it through as literal HTML.

2. **CELL content** — `cellsToMarkdownTable` in `tableCellMarkers.js` inserts
   cell text (only newlines collapsed, not escaped) verbatim into pipe-table cells:
   `| <script>alert(1)</script> |`. remark-gfm parses inline HTML inside table
   cells; rehype-raw renders it as-is.

The footnote transform (`$1` is `\d+` only) is safe — no model-controlled text
reaches the tag or attribute.

This risk is tracked for remediation in v3.1-05 (rehype-sanitize). The finding is
logged here as a warning to ensure the deferral is explicit in the review record.
It was not introduced by this phase but is surfaced by the new code paths that pipe
model output through MarkdownPreview inside FragmentCard.

**Planned fix (v3.1-05):** add `rehypeSanitize` (with a `defaultSchema` or
custom schema permitting `<sup>`, `<table>`, `<thead>`, `<tbody>`, `<tr>`, `<td>`,
`<th>`, `<b>`, `<i>`, `<em>`, `<strong>`) after `rehypeRaw`:
```jsx
<Markdown
  remarkPlugins={[remarkGfm]}
  rehypePlugins={[rehypeRaw, rehypeSanitize]}
  components={components}
>
```

## Info

### IN-01: Dead branch `i < 0` in `cleanTail` replacement callback

**File:** `src/modules/views/OnlyOffice/Scribe/assistantSegments.js:73`

**Issue:** Inside the tail-strip `replace` callback, the condition
`return i < 0 || i >= frags.length ? '' : m` includes `i < 0`. However, `i` is
derived from `parseInt(d, 10)` where `d` is captured by `\d+` — `parseInt` of a
non-empty digit string is always `>= 0`. The `i < 0` branch is permanently dead
code. This causes no incorrect behavior but is misleading.

**Fix:** Simplify to `return i >= frags.length ? '' : m`.
