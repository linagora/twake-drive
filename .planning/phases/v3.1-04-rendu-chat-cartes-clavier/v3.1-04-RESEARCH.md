# Phase v3.1-04: Rendu chat (cartes + clavier) - Research

**Researched:** 2026-06-19
**Domain:** React component rendering (in-bubble fragment cards), markdown-marker preprocessing, keyboard navigation / roving focus across a message thread — all inside the existing Scribe chat side panel.
**Confidence:** HIGH (entirely an in-repo refactor; every signature below was read from source, not assumed)

## Summary

This phase is a **pure front-end refactor inside one existing file tree** (`src/modules/views/OnlyOffice/Scribe/`). No new dependencies, no network, no backend. The work replaces the throwaway `composeAssistantDisplay` text-substitution helper (`ChatMessageList.jsx`) with real React `<FragmentCard>` components emitted at each `{{fragment:N}}` marker position, gives each card its own Copy/Insert/Replace buttons operating on the **raw** fragment string, extends `MarkdownPreview` to render plugin markers cosmetically, and wires arrow-key navigation that spans every card in the thread back to `ChatInput`.

The single most important constraint is **D-03 (the raw/display invariant)**: the rendered card content is cosmetically cleaned (REF → visible text, `[TABLE]`/`[CELL]` → pipe table, `[^scribe-fn-N]` → discreet marker), but the string passed to `panelActions.insert/replace` and to the clipboard must be the **untouched raw fragment** (markers intact). The existing reinjection pipeline (`View.jsx` → `castPanelAction` → OO plugin `code.js`) parses those markers to restore tables/images/footnotes/cross-refs. Cleaning the raw string before insert would silently break FRAG-03. The current code already honors a weaker version of this (it passes `msg.content` to `MessageActions`, never the composed display string) — the phase must preserve that discipline at fragment granularity.

**Primary recommendation:** Build a `<FragmentCard rawFragment hasSelection>` component (reusing `MessageActions`' three handlers verbatim — they already operate on a raw `content` string) plus a pure placement helper `buildAssistantSegments(discussion, fragments)` that returns an ordered array of `{type:'prose', md}` / `{type:'card', index, raw}` segments (the array-of-segments approach, replacing back-to-front text substitution). Render prose segments through the (extended) `MarkdownPreview` and card segments as `<FragmentCard>`. Keyboard nav: a single thread-level controller in `ChatMessageList` holding a flat ordered list of card refs + a `(cardIndex, buttonIndex)` focus coordinate, mirroring the manual-index pattern already proven in `ScribeActionMenu.jsx` and `ScribeResultPanel.jsx` (this repo does NOT use a roving-tabindex library — it uses manual index state + imperative `.focus()`).

### Pattern 1: Array-of-segments placement (replaces back-to-front substitution)
**What:** Convert `(discussion, fragments[])` into an ordered render list instead of one composed string.
**When to use:** Always, for assistant bubbles with `fragments.length > 0`.
**Why:** A `<Card>` is a React element, not text — it cannot be produced by string substitution. The current `composeAssistantDisplay` (ChatMessageList.jsx:32-73) already computes exactly the placement logic; lift it to emit segment objects.

```js
// Derived from composeAssistantDisplay (ChatMessageList.jsx:41-72) — same REF-safe placement,
// same "orphans at end", same out-of-range skip. Returns segments instead of a string.
// Source: in-repo extractChannelMarkers (scribeResponse.js:67)
export function buildAssistantSegments(discussion, fragments) {
  const disc = typeof discussion === 'string' ? discussion : ''
  const frags = Array.isArray(fragments) ? fragments : []
  if (frags.length === 0) return [{ type: 'prose', md: disc }]   // D-05/CONTRACT-02: no cards

  const hits = extractChannelMarkers(disc, 'fragment')           // ordered, 0-indexed, REF-safe
  const valid = hits.filter(h => h.index >= 0 && h.index < frags.length)
  const referenced = new Set(valid.map(h => h.index))

  const segments = []
  let cursor = 0
  for (const h of valid) {
    const marker = `{{fragment:${h.index}}}`
    const prose = disc.slice(cursor, h.position)
    if (prose) segments.push({ type: 'prose', md: prose })
    segments.push({ type: 'card', index: h.index, raw: frags[h.index] })
    cursor = h.position + marker.length
  }
  const tail = disc.slice(cursor)
  // strip any dangling out-of-range markers from the tail (mirrors lines 58-61)
  const cleanTail = tail.replace(/\{\{fragment:(\d+)\}\}/g, (m, d) => {
    const i = parseInt(d, 10); return i < 0 || i >= frags.length ? '' : m
  })
  if (cleanTail) segments.push({ type: 'prose', md: cleanTail })

  // D-04: orphan (unreferenced) fragments rendered as cards at the end
  frags.forEach((raw, i) => {
    if (!referenced.has(i)) segments.push({ type: 'card', index: i, raw })
  })
  return segments
}
```

### Pattern 3: Manual-index keyboard nav across the thread (mirror existing files)
**What:** A focus coordinate `(cardPos, buttonPos)` over a flat, thread-ordered card list.
**Proven precedent in this repo:**
- `ScribeActionMenu.jsx:42-103,165-424` — `focusIndex`/`submenuFocusIndex` state + `updateFocus()` + imperative `.focus()` via refs; ↑/↓ move index, ←/→ enter/leave, Enter activates, Escape closes. **No roving-tabindex lib.**
- `ScribeResultPanel.jsx:587-631` — `getFocusables()` returns an ordered ref array; one `handleKeyDown` computes `(currentIndex ± 1 + len) % len` and calls `items[next].focus()`. ←/→ and Tab cycle buttons.
- `ChatInput`↔`ScribeActionMenu` already model "arrow leaves the input": `ScribePromptInput.onArrow('up'|'down')` → `handlePromptArrow` (ScribeActionMenu.jsx:136-163). Reuse this exact shape for `ChatInput` ↑.

**Recommended mechanism (Claude's discretion per D-11 — confirm in plan):** manual index + refs (matches repo), NOT a roving-tabindex library.
- Each `FragmentCard` exposes its button refs (Copy/Insert/Replace) up to a controller in `ChatMessageList` via `forwardRef`/`useImperativeHandle` (the `useImperativeHandle` pattern is already used in `ScribeActionMenu.jsx:81-85`).
- Controller holds `cardList` = flat array in thread order (oldest→newest) and a focus coord.
- KBD-01 (↑ from input): focus **last** card's Insert button (D-10: walk backwards; skip pure-discussion messages with 0 cards). No wrap.
- KBD-02 (←/→): `buttonPos` cycles among present buttons (Replace only when `hasSelection`, D-07 — so the button array length is 2 or 3).
- KBD-03 (↑/↓): `cardPos` moves across all cards in the thread; ↓ past the most-recent card → `ChatInput.focus()`. No wrap upward past the oldest.
- KBD-04: Escape → `ChatInput.focus()`; Enter/Space → activate focused button (`<button onClick>` already fires on Enter/Space natively, but the controller intercepts arrows with `preventDefault`).
- D-09 focus ring: add `outline: 2px solid ${SCRIBE_PURPLE}` (or `:focus-visible`) on the focused button — `MessageActions` buttons currently have no outline; add one for the card variant.

### Anti-Patterns to Avoid
- **Cleaning raw before insert (D-03 violation):** never pass a marker-stripped string to `panelActions.insert/replace` or the clipboard. Breaks FRAG-03.
- **Hand-rolled `{{fragment:N}}` regex:** use `extractChannelMarkers` — a naive `\{\{.+?\}\}` would eat `{{REF:scribe-ref-N:…}}` (CONTRACT-04). The strict grammar is `FRAGMENT_MARKER_RE = /\{\{fragment:(\d+)\}\}/g` (scribeResponse.js:53).
- **Rendering an empty bubble:** D-05/CONTRACT-02 — 0 fragments → discussion-only, no card, no insert UI; empty discussion + fragments → render only the cards (mirror existing D-07 handling).
- **Roving-tabindex library:** inconsistent with the two existing manual-index implementations; adds a dependency out of scope.
- **Mutating `msg.fragments`:** the array is the stored raw model; cosmetic cleanup must be a local derived value only.

## Common Pitfalls

### Pitfall 1: Cleaning the fragment before insert (D-03)
**What goes wrong:** REF/table/footnote markers stripped from the string passed to `panelActions.insert` → OO inserts plain text, losing cross-refs/tables (FRAG-03 fails silently).
**Why it happens:** It's tempting to pass the same "nice" string you rendered.
**How to avoid:** Two separate values — `raw` (to actions/clipboard) and the cleaned display (computed inside `MarkdownPreview` from `raw`, never stored). `MessageActions content=raw`.
**Warning signs:** Inserted text shows literal `{{REF:...}}` is *good* (means raw reached the plugin); inserted text missing a cross-ref hyperlink means the marker was stripped before insert.

### Pitfall 2: Greedy fragment-marker regex eating cross-refs
**What goes wrong:** `{{REF:scribe-ref-2:Figure 1}}` consumed by a `\{\{.+?\}\}` substitution.
**How to avoid:** Only ever use `extractChannelMarkers(disc,'fragment')` / `FRAGMENT_MARKER_RE`. CONTRACT-04 has a preservation test; reuse the same helper that test guards.

### Pitfall 3: Flaky fragments (GATE §8) — render must not assume perfection
**What goes wrong:** Card render assumes a footnote/REF is always present; a non-deterministic tail occasionally drops `[^scribe-fn-N]` (1/6 on the edited-token case, GATE §8). Re-ask mitigation is **v3.1-05, out of scope here.**
**How to avoid:** Render whatever the raw fragment contains; never crash on a missing/extra marker. A fragment with no markers must still render as plain markdown. A `fellBack` response (`fragments: []`) must render discussion-only (CONTRACT-02). Treat the cosmetic preprocessor as best-effort — unmatched markers should pass through visibly rather than throw.

### Pitfall 4: Focus lost to the cross-origin OO iframe
**What goes wrong:** `ChatInput` already fights the OO iframe for focus on open (ChatInput.jsx:56-77, blur-and-retry burst). Returning focus to the input on Escape/↓-past-newest must use the same `textareaRef.current.focus()`, not rely on default tab order.
**How to avoid:** Expose an imperative `focus()` from `ChatInput` (forwardRef) the controller can call; mirror the existing one-shot focus discipline (do not start a reclaim loop).

### Pitfall 5: Cards inside a single bubble break maxWidth/scroll layout
**What goes wrong:** Current assistant bubble is `maxWidth:85%` with `<Markdown>` directly inside. Inserting bordered cards inline needs the bubble to host block-level children and tables (`overflowX:auto`) without clipping.
**How to avoid:** Render prose+cards as siblings inside the assistant container; `MarkdownPreview` already wraps tables in `overflowX:auto`. Auto-scroll-to-bottom (`ChatMessageList` useEffect:207-214) keys off `messages.length` — adding cards inside an existing message won't retrigger it, which is fine.

## Code Examples

### Extending MarkdownPreview for D-02 markers (cosmetic only)
```jsx
// Source: MarkdownPreview.jsx:134-141 (existing image-marker preprocessor) — extend the same
// string preprocessing stage. Order matters: tables first (they wrap multiline blocks).
const preprocessForDisplay = src => {
  if (typeof src !== 'string') return src
  let out = src
  // [TABLE:N]/[CELL:r,c] → GFM pipe-table (reuse existing pure helper)
  if (out.includes('[TABLE:') || out.includes('[CELL:')) {
    out = transformCellMarkersForPreview(out, '').displayMd   // enrichedMd '' → no count validation
  }
  // {{REF:scribe-ref-N:visible text}} → just the visible text  (REF_ID grammar from scribeProbe.js:104)
  out = out.replace(/\{\{REF:scribe-ref-\d+:([^}]*)\}\}/g, '$1')
  // {{IMG:scribe-img-N}} → existing image placeholder (already handled below this stage)
  out = out.replace(/\{\{IMG:(scribe-img-\d+)\}\}/g, '![IMG:$1](placeholder)')
  // [^scribe-fn-N] → discreet superscript marker (display only)
  out = out.replace(/\[\^scribe-fn-(\d+)\]/g, '<sup style="opacity:0.5">$1</sup>') // rehype-raw renders it
  return out
}
```
**Note:** `<sup>` raw HTML renders because `MarkdownPreview` already enables `rehypeRaw` (MarkdownPreview.jsx:156). Confirm in plan whether to keep the image-marker step in `MarkdownPreview` or factor all four into one helper — D-02 leaves the implementation to Claude's discretion, provided D-03 (raw untouched) holds. The preprocessor receives `raw` and returns a *new* string; `raw` itself is never modified.

## Security Domain

`security_enforcement` not explicitly `false` in config → included. This phase renders **LLM-produced text** (attacker-influenceable) into the DOM, so input-handling and injection are the relevant categories.

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | — (no auth surface) |
| V3 Session Management | no | — |
| V4 Access Control | no | — |
| V5 Input Validation / Output Encoding | **yes** | LLM text already passes `parseScribeResponse` (prototype-pollution-safe, DoS-bounded, ReDoS-safe — scribeResponse.js:22-31). Render via `react-markdown` which escapes by default. **Caution:** `MarkdownPreview` enables `rehypeRaw` (raw HTML passthrough). The D-02 footnote `<sup>` injection relies on this — but it means model-supplied raw HTML in a fragment is also rendered. |
| V6 Cryptography | no | — |

### Known Threat Patterns for {react-markdown + rehype-raw}
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS via raw HTML in LLM fragment (rehype-raw renders `<script>`/`<img onerror>`) | Tampering / Elevation | `[ASSUMED]` Pre-existing exposure — `MarkdownPreview` already uses `rehypeRaw` today (MarkdownPreview.jsx:156) for the popover path, so this phase does not introduce it. Recommend the plan confirm whether a `rehype-sanitize` pass is warranted given fragments are now rendered in chat too. Do NOT regress by removing existing escaping. Flag for human decision — out-of-scope hardening may belong to v3.1-05. |
| Prototype pollution via crafted JSON | Tampering | Already mitigated upstream (`Object.create(null)` staging, scribeResponse.js:307) |
| ReDoS via marker regexes | DoS | All marker regexes are linear (fixed literal + `\d+`); the new preprocessor regexes (`{{REF:…}}`, `[^scribe-fn-\d+]`) must stay linear — no nested quantifiers |

**Security note for planner:** the only *new* injection vector this phase could add is the cosmetic `<sup>` HTML and any future raw-HTML-emitting preprocessor step. Keep preprocessor output to controlled, fixed HTML (e.g. a `<sup>` with inline style around a captured `\d+`), never interpolate model-controlled substrings into an HTML attribute/tag without encoding.
