# Phase v3.1-04: Rendu chat (cartes + clavier) - Context

**Gathered:** 2026-06-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Dans le panneau **chat** (pas le popover — c'est v3.1-05), rendre chaque fragment d'une réponse
assistant dans une **carte encadrée distincte**, positionnée à l'emplacement de son marqueur
`{{fragment:N}}` dans la discussion, avec les actions **Copier / Insérer / Remplacer**, le tout
entièrement pilotable au **clavier** comme à la souris. Une réponse de pure discussion (0 fragment)
n'affiche **aucune carte ni UI d'insertion**.

Le *quoi* est verrouillé par 9 requirements (CONTRACT-02, FRAG-01..04, KBD-01..04). Cette phase
décide le *comment* (apparence, rendu des marqueurs, disposition, affordance clavier) et remplace le
throwaway `composeAssistantDisplay` de v3.1-02.

**Hors scope (→ v3.1-05) :** rendu popover du fragment unique, re-ask sur parse invalide, i18n des
libellés/repli sur 5 locales, corpus de régression, décision du flag `response_format`.
</domain>

<decisions>
## Implementation Decisions

### Apparence de la carte & rendu du fragment
- **D-01:** Carte **encadrée** (bordure fine + léger fond, accent violet Scribe `#7C3AED` déjà
  utilisé dans `ChatMessageList`), contenu du fragment **rendu en markdown riche** (gras, listes,
  tableaux) — pas la source brute. Réutiliser le rendu Markdown existant (react-markdown /
  `MarkdownPreview`). Pas de toggle source en v3.1-04.

### Marqueurs plugin dans la carte
- **D-02:** Les marqueurs plugin présents dans un fragment sont **rendus proprement** à l'affichage :
  `{{REF:scribe-ref-N:texte}}` → afficher uniquement le *texte visible* ; `[TABLE:N]/[CELL:r,c]` →
  tableau rendu ; `[^scribe-fn-N]` → marqueur de note discret. Étendre `MarkdownPreview` (qui
  pré-traite déjà les marqueurs d'images) pour couvrir ces marqueurs.
- **D-03:** ⚠️ INVARIANT CRITIQUE — l'affichage « propre » est **purement cosmétique**. Les actions
  Copier/Insérer/Remplacer opèrent sur le **fragment BRUT** (marqueurs intacts), jamais sur la
  version affichée nettoyée. C'est ce que le pipeline de réinjection (FRAG-03) attend pour préserver
  tables/images/footnotes/cross-refs. Séparer strictement `displayContent` (cosmétique) et
  `rawFragment` (passé à `panelActions.insert/replace` et au presse-papier).

### Disposition discussion + cartes
- **D-04:** **Une seule bulle assistant** : la prose de discussion rendue, avec les cartes insérées
  **inline à la position de leur marqueur** `{{fragment:N}}`. Les fragments **orphelins** (non
  référencés par un marqueur) sont rendus en **fin de bulle**, après la prose. (Logique de
  positionnement déjà esquissée par `composeAssistantDisplay` via `extractChannelMarkers` — la
  réutiliser pour le placement, mais émettre des composants `<Card>` au lieu d'une substitution
  texte.)
- **D-05:** CONTRACT-02 — une réponse à **0 fragment** rend la discussion seule, **sans aucune carte
  ni bouton d'insertion**. Pas de bulle vide non plus (cf. comportement D-07 existant).

### Actions par carte (réutilisation MessageActions)
- **D-06:** Réutiliser la logique de `MessageActions` **par carte** : Copier (presse-papier
  markdown+HTML via `markdownToHtml` + `stripParagraphWrapper`), Insérer (`panelActions.insert`),
  Remplacer (`panelActions.replace`), confirmation **check vert 1,5 s**. Chaque carte porte ses
  propres boutons opérant sur SON fragment brut.
- **D-07:** FRAG-04 — le bouton **Remplacer n'apparaît que si une sélection est active** dans le
  document (réutiliser le gating `hasSelection`/`currentSelection` déjà en place).
- **D-08:** L'ancien `MessageActions` au niveau message (qui agit sur `msg.content` entier) est
  **retiré** du rendu assistant : les actions vivent désormais uniquement sur les cartes de fragment.

### Affordance focus clavier & cas limites (KBD-01..04)
- **D-09:** Indicateur de focus = **anneau visible** (outline accent) sur le bouton focalisé.
- **D-10:** « Carte la plus récente » (KBD-01, ↑ depuis l'input) = la dernière carte du **fil** en
  remontant. Si le dernier message est de la **pure discussion** (0 carte), **remonter** au dernier
  fragment du message précédent qui en contient. **Pas de wrap.** ↓ au-delà du fragment le plus
  récent **revient à l'input** (KBD-03).
- **D-11:** Navigation conforme aux reqs : ←/→ cyclent Copier/Insérer/Remplacer sur la carte
  focalisée (KBD-02) ; ↑/↓ parcourent les fragments à travers le fil (KBD-03) ; Échap → input ;
  Entrée/Espace activent le bouton focalisé (KBD-04). Approche suggérée : roving tabindex sur les
  boutons des cartes (laisser le planner confirmer le mécanisme précis).

### Claude's Discretion
- Mécanisme technique exact de la nav clavier (roving tabindex vs gestion manuelle d'index/refs),
  structure des composants (un `<FragmentCard>` dédié + un parseur de placement), nommage.
- Détails de style (rayons, espacements, épaisseur de bordure) tant que la carte reste « encadrée,
  visuellement distincte de la discussion » (FRAG-01) et cohérente avec le thème clair/sombre.
- Implémentation du rendu propre des marqueurs (extension de `MarkdownPreview` vs petit pré-processeur
  dédié), tant que D-03 (raw préservé pour les actions) est respecté.
</decisions>
