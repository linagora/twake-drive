# Phase v3.1-04 Plan 03: FragmentCard (cartes + clavier substrate) Summary

Bordered Scribe-purple `FragmentCard` that displays a raw LLM fragment through the Plan-02 cosmetic `MarkdownPreview` while carrying its own Copy/Insert/Replace actions that operate on the verbatim RAW fragment (markers intact) — the D-03 invariant the OO rich-reinjection pipeline depends on.

## What Was Built

- **Task 1 — MessageActions focus ring (D-09):** Added a local `SCRIBE_PURPLE` accent and `onFocus`/`onBlur` inline-style handlers that set/clear a `2px solid #7C3AED` outline on the focused action button. Handler logic, the `hasSelection` gate, and the 1500ms green-check confirmation are unchanged; buttons remain plain DOM-queryable `<button>` elements with i18n aria-labels — the focusable substrate Plan 05's keyboard controller will drive.
- **Task 2 — Failing FragmentCard spec (RED):** RTL spec mocking `useTheme`, `Tooltip`, `twake-i18n`, and `useScribe` (with `insert`/`replace` jest spies). Covers FRAG-02 (3 buttons), FRAG-04 (Replace gated off without selection), the D-03 raw-arg guard (insert/replace receive the verbatim raw with `{{REF:...}}` and `[^scribe-fn-2]` substrings present), the display-vs-raw split (visible "Fig 1" shown while action gets the raw marker), and D-01 (Scribe-purple border on the card root).
- **Task 3 — FragmentCard implementation (GREEN):** Named-export functional component rendering a bordered, theme-aware card (`1px solid #7C3AED`, purple fill, radius 8) containing `<MarkdownPreview>{raw}</MarkdownPreview>` for cosmetic display and `<MessageActions content={raw} hasSelection={hasSelection} />` for actions. No marker-cleaning is done in FragmentCard — `raw` flows untouched to both consumers (D-03).
