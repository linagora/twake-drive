# Phase v3.1-02 Plan 02: Chat seam, message model, render Summary

Wired the v3.1-01 response contract through the CHAT surface end-to-end: the chat
system prompt now emits the shared `{ discussion, fragments[] }` JSON contract, every
raw LLM response is parsed via `parseScribeResponse(raw, { surface: 'chat' })` before
storage, the assistant message model carries `discussion`/`fragments`/`fellBack` while
keeping `content == discussion`, multi-turn history serializes assistant turns via
`serializeAssistantTurnForHistory`, and a throwaway render-time helper composes
`discussion` + `fragments` for display with zero functional regression.

## What Changed

### Task 1 — ScribeContext.jsx (commit 08cdd6693)
- **CHAT_SYSTEM_PROMPT (D-01/D-02 chat side):** extended to describe the `{ discussion, fragments[] }` contract with a single tiny worked example showing a `{{fragment:0}}` marker, plus the chat cardinality sentence (0..N fragments; `discussion` may be free-form). Kept the existing same-language / use-Markdown guidance and the real `system` role.
- **Parse seam (D-13):** imported `parseScribeResponse` and `serializeAssistantTurnForHistory`. After `callScribeAI(...)` returns the raw string, `parseScribeResponse(responseText, { surface: 'chat' })` runs before any `setMessages` of the assistant turn.
- **Extended model (D-10):** the stored assistant message sets `content: parsed.discussion` (content == discussion preserved) and adds `discussion`, `fragments`, `fellBack`. User and error message shapes unchanged.
- **History serialization (D-12):** the `aiMessages` map's assistant-turn branch now calls `serializeAssistantTurnForHistory({ discussion: m.discussion ?? m.content, fragments: m.fragments })` instead of verbatim `content: m.content`. User-turn mapping (including the selection-composite branch) and the system message are untouched. `callScribeAI`, the error path, and loading/abort logic are unchanged.

### Task 2 — ChatMessageList.jsx (commit 2385af159)
- Added `composeAssistantDisplay(discussion, fragments)` — a render-time helper annotated as throwaway (replaced by fragment cards in v3.1-04):
  - Replaces each `{{fragment:N}}` marker in `discussion` with `fragments[N]` (0-indexed), substituting back-to-front so positions stay valid. Uses the REF-safe `extractChannelMarkers` (no hand-rolled marker regex).
  - Appends unreferenced fragments at the end, blank-line separated (D-04).
  - **D-06:** with `fragments == []` it returns `discussion` byte-identically (the chat fallback `{ discussion: raw, fragments: [] }` renders raw exactly as today).
  - **D-07:** empty `discussion` + non-empty fragments returns just the joined fragment(s) — no empty/broken bubble, no synthesized labels.
- The assistant branch composes display text only when `discussion`/`fragments` are present (`hasContractFields`), falling back to `msg.content` for legacy messages. Stored content is never mutated. `MessageActions` still receives `msg.content` (pure discussion channel, D-05).

## Deviations from Plan

None — plan executed exactly as written.

## No-Regression Proof

- **content == discussion invariant:** assistant message `content` is set to `parsed.discussion`; structured fields are purely additive.
- **Fallback byte-identity (D-06):** verified by reasoning + a logic reproduction — `composeAssistantDisplay(raw, [])` returns `raw` unchanged, so on parse fallback (`{ discussion: raw, fragments: [] }`) the bubble renders raw byte-identically to today.
- **Empty-discussion (D-07):** `composeAssistantDisplay('', ['only frag'])` returns `'only frag'` (no empty bubble).
- **Marker substitution / orphans / multi-marker out-of-order:** all verified (`'a {{fragment:0}} b'` + `['X']` -> `'a X b'`; `'hello'` + `['frag1']` -> `'hello\n\nfrag1'`; `'{{fragment:1}} mid {{fragment:0}}'` + `['A','B']` -> `'B mid A'`).
- **Contract layer untouched:** existing `scribeResponse.spec.js` — 38/38 tests pass.

## Threat Surface

No new surface introduced. All raw responses pass through `parseScribeResponse` (never
throws, DoS-bounded, prototype-pollution safe — v3.1-01). The compose helper reuses the
strict REF-safe `extractChannelMarkers` so it cannot match/alter `{{REF:scribe-ref-N:…}}`
markers. Assistant history uses `serializeAssistantTurnForHistory`, which truncates and
never emits full fragment bodies or raw contract JSON. Rendering stays on the existing
react-markdown path (no new `dangerouslySetInnerHTML`).

## Known Stubs

`composeAssistantDisplay` is an intentional, documented throwaway helper. It reproduces
today's single-blob rendering and is explicitly replaced by real fragment cards in
v3.1-04 (annotated in code and per phase plan D-04). Not a blocking stub — it is the
no-regression rendering strategy for this phase.
