# Phase v3.1 Plan 02-01: Inline contract prompt Summary

Inline popover prompt now instructs the model to return the v3.1-01
`{ discussion, fragments[] }` JSON contract with **exactly ONE** fragment,
anchored by a single `{{fragment:0}}` worked example — a prompt-text-only,
no-regression change to `scribeAI.js`.

## What Was Built

### Task 1 — Shared contract block + inline cardinality (commit a8dddebbd)
- Added `RESPONSE_CONTRACT_BLOCK` exported constant (D-01): terse plain-language
  rules describing the `{ discussion, fragments[] }` JSON contract, anchored by
  ONE tiny worked example showing a `{{fragment:0}}` marker resolving to
  `fragments[0]`. No few-shot, no multiple examples.
- Wired the block into `buildMessages`: `systemBase` now concatenates
  `SYSTEM_PROMPT + RESPONSE_CONTRACT_BLOCK + ' Return **exactly ONE** fragment.'`
  (D-02, inline side) BEFORE the conditional marker appendices and before
  `systemPrefix = systemBase + '\n\n'`.
- Stays entirely in the single user-role content prefix — **no system role
  introduced**.

### Task 2 — Reword conditional marker rules (commit 4cbf59418)
- The three existing conditional appendices (TABLE/CELL, `[^scribe-fn-`, `{{REF:`)
  now each begin with "Inside the fragment string(s) (never in `discussion`), …"
  (D-03), framing the markers as belonging inside fragments per
  `SCRIBE_OUTPUT_SCHEMA.md`.
- Each block remains **strictly conditional** on its marker literal appearing in
  `extra.enrichedMd` — trigger conditions unchanged, no block made unconditional.
- The REF rule keeps its allowance to modify visible text while never altering
  the `scribe-ref-N` identifier.
