## Summary

The plumbing phase wires `parseScribeResponse` into both surfaces (chat via `ScribeContext`,
popover via `ScribePopover`) and adds the compose helper in `ChatMessageList`. The core
intent is correct and the parse seam is placed at the right boundary in both paths. Two

## Critical Issues

### CR-01: `composeAssistantDisplay` silently drops all fragments when `discussion` is empty but markers exist

**File:** `src/modules/views/OnlyOffice/Scribe/ChatMessageList.jsx:59-61`

**Issue:** When `disc` (the substituted-discussion string) has length 0 after all
`{{fragment:N}}` markers are replaced in-place, the function short-circuits to
`orphanText` — but at that point `referenced` already contains the indices that were
substituted, so the "orphans" filter removes them from `orphanText`. The result: all
successfully-referenced fragments were spliced into `composed` which shrinks to `""`,
and then every non-orphan is excluded from `orphanText`. The returned string is `""`.

Concrete trigger: `discussion = "{{fragment:0}}"`, `fragments = ["the text"]`. After
back-to-front substitution `composed` becomes `"the text"` (length 9, not 0), so this
branch is not hit in the nominal case. However if discussion is purely whitespace that
is coerced away earlier (e.g. `"  "`) then disc starts as `"  "`, no markers are found
by `extractChannelMarkers`, zero substitutions occur, `composed` remains `"  "` (length
2), and the orphan path is taken — all fragments appear as orphans and `orphanText` is
returned. That is actually correct for the whitespace case.

The genuine bug fires when `discussion = ""` AND `fragments` is non-empty and all
non-empty. In that case the D-06 guard at line 37 (`if (frags.length === 0) return disc`)
does NOT fire (frags is non-empty). The loop finds zero hits (disc has no markers), so
`referenced` is empty, every fragment is an orphan, and `orphanText = frags.join('\n\n')`.
Then `composed.length === 0` is true, so the function returns `orphanText` — which is
correct. But there is a subtler variant: if `discussion = ""` and the LLM emitted
`{{fragment:0}}` but the discussion field came back empty due to fellBack trim — in that
case there are no markers to locate, all fragments are orphans, and the result is correct.

Re-examining: the only genuinely broken path is when ALL markers that were hit in disc
are _out-of-range_ (filtered by `if (index < 0 || index >= frags.length) continue`).
When that happens, `referenced` stays empty, but `composed` is still `disc` with its
original `{{fragment:N}}` marker text still present (they were not replaced). The
orphanText includes all fragments, so the raw markdown markers literally appear in the
rendered output alongside the duplicate fragment text appended at the bottom.

**Fix:** After the substitution loop, for any hit that was skipped (out-of-range), leave
the marker as-is in `composed` (current behavior) — but the orphan calculation already
handles this correctly because those indices are never added to `referenced`. The real
issue is marker text not being stripped. Add a second pass to strip any remaining
`{{fragment:N}}` markers where index is out of range:

```js
// After the substitution loop, strip dangling out-of-range markers from composed.
composed = composed.replace(/\{\{fragment:(\d+)\}\}/g, (m, d) => {
  const idx = parseInt(d, 10)
  return (idx < 0 || idx >= frags.length) ? '' : m
})
```

Or, simpler for a throwaway helper: run a single regex replace after substitution to
remove any remaining `{{fragment:\d+}}` tokens that were not resolved.

### CR-02: `handleActionSelect` dep array omits `enrichedMd` and `tableAmbiguity` — stale closure on retry

**File:** `src/modules/views/OnlyOffice/Scribe/ScribePopover.jsx:209`

**Issue:** The `useCallback` dep array is:
```js
[selectedText, selectedHtml, client, t, addMessage]
```
Both `enrichedMd` (lines 99, 106, 117, 139, 140, 166) and `tableAmbiguity` (line 96)
are closed over inside the callback but missing from the dep array.

`tableAmbiguity` guards the entire callback body at line 96. If the parent re-renders
with `tableAmbiguity` going from non-null to null (ambiguity resolved), the stale
closure still sees the old truthy value and silently swallows the action, making the
popover appear broken.

`enrichedMd` is used to build the actual prompt (`extra.enrichedMd`, line 140) and the
preview transform (`transformCellMarkersForPreview(normalizedFragment, enrichedMd)`,
line 166). If the prop changes between renders (e.g. the plugin sends a richer extraction
after the first render), retries fire with the old `enrichedMd`, producing a prompt
mismatch and a mismatched cell-marker preview. More importantly, on retry via
`handleRetry → handleActionSelect`, the enrichedMd used for the LLM call and the one
used for the cell-marker display check will be the stale version.

**Fix:**
```js
}, [selectedText, selectedHtml, enrichedMd, tableAmbiguity, client, t, addMessage])
```

### WR-01: Chat history serialization passes error messages through `serializeAssistantTurnForHistory`

**File:** `src/modules/views/OnlyOffice/Scribe/ScribeContext.jsx:139-147`

**Issue:** The history map filters `m.role !== 'error'` at line 125, which correctly
excludes error messages from the LLM context. However the `if (m.role === 'assistant')`
branch at line 139 only matches `assistant` turns, so error messages that somehow slip
through (or future code that changes the filter) fall through to the generic
`return { role: m.role, content: m.content }` path. This is not a bug in the current
code because the filter appears at line 125 before the map. The structural concern is
the defense-in-depth gap: the assistant serializer is tested by role match alone — if
the filter were ever removed or re-ordered, error messages with `role: 'error'` would
be forwarded to the LLM as-is. Additionally, the `role: 'error'` value is a UI-only
role that no LLM API accepts; it would cause a 400 from the stack's AI endpoint.

**Fix:** Add an explicit guard at the start of the map to drop non-`user`/`assistant`
turns, independent of the outer filter:

```js
.filter(m => m.role === 'user' || m.role === 'assistant')
.map(m => { ... })
```

This makes the serialization path self-contained and safe against future filter drift.

### WR-02: `hasContractFields` heuristic fires on messages with only `discussion` set, passing `undefined` fragments to `composeAssistantDisplay`

**File:** `src/modules/views/OnlyOffice/Scribe/ChatMessageList.jsx:235-238`

**Issue:** The guard is:
```js
const hasContractFields =
  msg.discussion !== undefined || msg.fragments !== undefined
```
A message that has `discussion` but no `fragments` property (e.g. an older mirrored
turn or a manual `addMessage` call) satisfies the OR condition and enters the compose
path. `composeAssistantDisplay` is called as:
```js
composeAssistantDisplay(msg.discussion ?? msg.content, msg.fragments)
```
When `msg.fragments` is `undefined`, the second argument is `undefined`. Inside
`composeAssistantDisplay`, line 34 guards this:
```js
const frags = Array.isArray(fragments) ? fragments : []
```
So `undefined` is coerced to `[]` and the D-06 fast path returns `disc` unchanged.
This is safe in practice, but the heuristic is semantically incorrect: a message with
only `discussion` set should still use `msg.content` as the display source (the
pre-contract path), not `msg.discussion`. If `discussion` somehow differs from
`content` in such a message (forward-compat scenario), the display will silently diverge.

**Fix:** Require BOTH fields to be present before entering the compose path:
```js
const hasContractFields =
  msg.discussion !== undefined && msg.fragments !== undefined
```
This matches the shape of every extended-model message produced by this phase (both
`ScribeContext` and `ScribePopover` always set both).

### WR-03: Popover mirrors `parsed.discussion` as `content` for the assistant turn; chat compose helper uses `msg.discussion ?? msg.content` — the two diverge when `discussion` is empty and `fellBack` is false

**File:** `src/modules/views/OnlyOffice/Scribe/ScribePopover.jsx:183` /
`src/modules/views/OnlyOffice/Scribe/ChatMessageList.jsx:238`

**Issue:** For a well-formed popover response with `discussion = ""` and
`fragments = ["..."]` (a valid contract — no discussion, just a fragment):
- `content` is set to `""` (line 183)
- The compose helper path is entered because `hasContractFields` is true
- `composeAssistantDisplay("", ["..."])` returns `"..."` (the fragment text), correct

But `MessageActions` reads `msg.content` directly (line 243), which is `""`. If
`MessageActions` has any "insert" or "copy" action that forwards `content` to the
editor, the empty string is forwarded instead of the composed fragment text.

This is a direct consequence of the D-05 spec ("content == discussion") colliding with
the popover's valid empty-discussion turns. The THROWAWAY comment acknowledges this
will be resolved in v3.1-04, but the current behavior is silently wrong for inline
popover turns that produce a response with no discussion text.

**Fix (for this phase):** When mirroring inline turns in ScribePopover, set `content`
to `normalizedFragment` instead of `parsed.discussion`, preserving today's behavior for
`MessageActions` exactly:
```js
addMessage({
  id: Date.now() + 1,
  role: 'assistant',
  content: normalizedFragment,   // ← replaces parsed.discussion
  discussion: parsed.discussion,
  fragments: parsed.fragments,
  fellBack: parsed.fellBack,
  timestamp: new Date()
})
```
This keeps `content` as the "single insertable blob" for existing consumers while
`discussion`/`fragments` carry the structured form for v3.1-04.
