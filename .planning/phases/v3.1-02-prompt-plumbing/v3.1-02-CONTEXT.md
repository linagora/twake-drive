## Phase Boundary

Wire the v3.1-01 contract (`parseScribeResponse`) through both live surfaces so each one **emits** the `{discussion, fragments}` contract in its system prompt and **parses** the response at reception:

- **Inline popover** — `ScribePopover.handleActionSelect` → `buildMessages` / `callScribeAI` (in `scribeAI.js`)

Any display composition introduced here is a throwaway helper, replaced by real fragment cards in v3.1-04.

<decisions>
## Implementation Decisions

### Contract prompt design
- **D-01:** Prompt style = **terse rules + ONE worked example**. Plain-language contract description ("Return ONLY a JSON object with `discussion` and `fragments[]`…") anchored by a single tiny example showing a `{{fragment:N}}` marker. No few-shot. Best reliability-per-token for a generic model behind the cozy-stack proxy (no native `json_object` mode in v3.1).
- **D-02:** Cardinality difference expressed via **one shared contract block + a surface-specific sentence** (DRY, single contract to maintain):
  - inline → "Return **exactly ONE** fragment."
  - chat → "Return **0..N** fragments; `discussion` may be free-form."
- **D-03:** Marker-preservation rules (TABLE/CELL/footnote/REF) stay **conditional** (only appended when the corresponding marker is present in `enrichedMd`, as today in `buildMessages`), but **reworded to state the rules apply INSIDE fragment strings** — per `SCRIBE_OUTPUT_SCHEMA.md`, these markers belong in `fragments`, never in `discussion`. Preserves token economy; no unconditional bloat.

### Interim chat display (before v3.1-04 cards)
- **D-04:** Chat assistant turns are displayed via a **render-time compose helper**: build display text = `discussion` with each `{{fragment:N}}` marker replaced by `fragments[N]` (unreferenced fragments appended at end). Visually ~identical to today's single-blob response → strongest no-regression proof. The helper is **throwaway**, deleted when fragment cards land in v3.1-04.
- **D-05:** The compose helper lives at **render time** (in the existing chat message component), NOT in stored `content`. The message model keeps **`content == discussion` (pure)** + `fragments[]` + `fellBack` as structured fields. This honors criterion 3 literally and keeps multi-turn serialization reading the `discussion` field untouched.
- **D-06:** Fallback degrades **naturally**: on parse failure the chat fallback yields `{discussion: raw, fragments: []}`, so the compose helper renders `raw` byte-identically to today. No special-casing.
- **D-07:** Empty-discussion turns (common for mirrored inline turns) are handled by the **same compose helper** — it shows just the fragment(s) when `discussion` is empty (no marker, fragment appended), so no empty/broken bubble appears. No synthesized labels.

### Inline cardinality normalization (INLINE-01)
- **D-08:** Inline normalizes a validly-parsed response to exactly one insertable unit (re-ask is deferred to v3.1-05):
  - **2+ fragments** → **join with `\n\n`** into one insertable blob (matches today's "whole response is insertable"; no content loss).
  - **0 fragments** → **promote `discussion`** to the single fragment (inline's purpose is to produce insertable text).
  - **1 fragment** → use directly.
- **D-09:** Inline shows/inserts only the (normalized) **fragment** — `discussion` is NOT rendered in the popover (INLINE-01). The existing cell-marker preview transform (`transformCellMarkersForPreview`) and reinjection apply to the normalized fragment text (replacing today's `rawResult`).

### Message model + mirroring (INLINE-02)
- **D-10:** Extended message model = existing fields **plus** `{ discussion, fragments, fellBack }`, with **`content == discussion`** retained. Existing UI continues to read `content` unchanged.
- **D-11:** Inline exchanges mirror the **full parsed turn** into shared chat history (`{ content: discussion, fragments, fellBack }` for the assistant turn, plus the user/breadcrumb turn). When the side panel opens, the render-time compose helper renders inline turns identically to chat-native turns — and v3.1-04 cards will light up for inline turns too.
- **D-12:** Multi-turn LLM history is built via the existing **`serializeAssistantTurnForHistory(parsed)`** (v3.1-01): sends `discussion` + a truncated compact fragment note, never raw fragment bodies. Replaces the current verbatim `m.content` mapping for assistant turns in `ScribeContext.sendMessage`'s `aiMessages` construction. Honors criterion 4 exactly with tested code.

### Parse seam placement
- **D-13:** Both call sites route the raw LLM response through `parseScribeResponse(raw, { surface })` **before any display or storage**: `surface: 'chat'` in `ScribeContext.sendMessage`, `surface: 'popover'` (inline) in `ScribePopover.handleActionSelect`. This is the single seam required by criterion 2.

### Claude's Discretion
- Exact wording/whitespace of the contract instruction blocks and the worked example (subject to D-01/D-02/D-03 shape).
- Naming/location of the throwaway render-time compose helper.
- Whether `content` stores discussion with `{{fragment:N}}` markers stripped or retained (markers are invisible once the helper composes; either is acceptable as long as `content` is the pure discussion channel).

</decisions>

## Existing Code Insights

### Reusable Assets
- `parseScribeResponse` / `serializeAssistantTurnForHistory` (v3.1-01): the entire contract layer is built and tested — this phase only wires it in. No new parsing/serialization logic.
- `transformCellMarkersForPreview(text, enrichedMd)` (used in `ScribePopover`): keeps working; now applied to the normalized inline fragment instead of the raw response.
- `addMessage` (ScribeContext): already used by `ScribePopover` to mirror inline turns — extend the mirrored payload to the new model shape.

### Established Patterns
- Inline prompt is prepended into a **single `user` role** (no `system` role) to support RAG backends — preserve this when adding the contract block in `buildMessages` (D-01/D-02 must live in the user-content prefix).
- Chat uses a real `system` role (`CHAT_SYSTEM_PROMPT`) — contract block added there.
- `callScribeAI` returns a raw string; parse happens at the call site (D-13), keeping `callScribeAI` unchanged.

### Integration Points
- `ScribeContext.sendMessage` → `aiMessages` map: the only place assistant history is serialized for the LLM (D-12).
- `ScribePopover.handleActionSelect` → `addMessage`: the inline→chat mirror seam (D-11).
