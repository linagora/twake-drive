<design_decision>
The CONTEXT integration note (D-04) left the helper shape "à trancher au plan". DECISION — Option B (standalone exported wrapper), NOT folding retry into callScribeAI:

  export async function callScribeAIWithReask(client, messages, { signal, surface } = {})
    -> Promise<{ discussion, fragments, valid, fellBack, warnings, raw }>

Rationale:
- callScribeAI stays a pure single-shot transport (still independently callable + already unit-shaped); the retry+parse POLICY lives in one named seam.
- Returning the PARSED contract object (not raw text) is the single shared seam both call sites want: today each site calls callScribeAI THEN parseScribeResponse(surface) — collapsing both into one helper is what guarantees chat/popover cannot diverge.
- Exactly one helper invoked at both sites satisfies D-04 (no divergence) and D-05 (one retry, signal-respecting) structurally.

Re-ask trigger (D-01): first.fellBack === true OR first.valid === false. Both already exist on the parse result — NO new detection.
Re-ask mechanic (D-03): the second call appends ONE corrective message (REASK_CORRECTION_NUDGE) to the ORIGINAL messages array, NOT a blind retry of the same messages.
</design_decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM/proxy -> client | The model response (incl. the re-ask response) is untrusted, attacker-influenceable text crossing into the parser. |
| caller -> helper | AbortController signal is the user's cancel intent crossing into the transport. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-v3.1-06-01 | Denial of Service | callScribeAIWithReask retry loop | mitigate | Hard cap of exactly ONE re-ask (D-05): the second parse is returned unconditionally, no recursion, no `while`. A malicious/looping model can never cause >2 transport calls. Asserted by the call-count spec. |
| T-v3.1-06-02 | Tampering | re-ask response text | mitigate | Second response flows through the SAME parseScribeResponse (prototype-pollution-safe Object.create(null) staging, ReDoS-safe linear regexes, MAX_RAW_LENGTH guard) — no new parse path bypasses existing hardening. Invariant preserved by construction (helper only calls parseScribeResponse). |
| T-v3.1-06-03 | Denial of Service | abort handling on 2nd call | mitigate | The caller's `signal` is forwarded to BOTH callScribeAI calls; AbortError propagates (not swallowed), so a cancelled request cannot hang on the re-ask. Asserted by the abort spec. |
| T-v3.1-06-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (helper uses only existing callScribeAI + parseScribeResponse). No legitimacy gate required. |
