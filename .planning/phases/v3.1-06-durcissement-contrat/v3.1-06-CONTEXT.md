<domain>
## Phase Boundary

Durcir le contrat de réponse structurée pour un déploiement fiable, **en logique pure (aucune UI OO)**. Trois livrables :
1. **Re-ask** une seule fois sur réponse au parse invalide, avant d'appliquer le repli contextuel (HARDEN-01).
2. **Corpus de régression** de réponses malformées + tests de cas limites au vert ; re-mesure du taux de `fellBack` code-fence (HARDEN-02).
3. **Décision documentée** du défaut du flag `response_format` (HARDEN-02).

Hors scope : tout rendu/UI (cartes, popover, menu) — couvert par v3.1-04/05 ; i18n des libellés — phase v3.1-07.
</domain>

<decisions>
## Implementation Decisions

### Déclencheur du re-ask (HARDEN-01)
- **D-01:** Le re-ask se déclenche si `fellBack === true` (JSON illisible) **OU** `valid === false` (split-table). Ces deux signaux existent déjà dans le retour de `parseScribeResponse` — pas de nouvelle détection à construire.
- **D-02:** La brèche d'intégrité REF/footnote flaky (citée dans STATE) n'entre **pas** dans le déclencheur de cette phase (elle demanderait une comparaison marqueurs entrée/sortie au call site, hors signal actuel du contrat). À reconsidérer ultérieurement si le tail REF persiste.

### Mécanique et surfaces du re-ask
- **D-03:** Re-ask **correctif** : le 2e appel ajoute un message correctif (instruction « réponds UNIQUEMENT en JSON conforme au contrat ») plutôt qu'un simple retry des mêmes messages.
- **D-04:** Appliqué aux **deux surfaces** — chat (ScribeContext.jsx:198) ET popover/inline (ScribePopover.jsx:220) — via un **helper partagé** (côté `scribeAI.js`, près de `callScribeAI`) pour éviter la divergence.
- **D-05:** **Un seul** retry max avant repli. Respect strict de `signal`/abort pendant le 2e appel ; le re-ask ne doit pas casser la gestion du loading ni doubler les comptages.

### Défaut response_format (HARDEN-02)
- **D-06:** **Garder** `response_format: { type: 'json_object' }` (scribeAI.js:279) comme défaut. La décision et la contrainte du proxy cozy-stack sont **documentées** ; `json_schema` (structured-output, via `SCRIBE_OUTPUT_SCHEMA`) reste une option future documentée, non activée dans cette phase.

### Corpus de régression (HARDEN-02)
- **D-07:** **Module de fixtures statiques + tests Jest déterministes** sur `parseScribeResponse`, **séparé** du harness GATE (qui mesure le modèle live). Catégories couvertes : fence (```json), préambule, virgule traînante, split-table, REF cassée, prose non-JSON.
- **D-08:** Re-mesure du taux `fellBack` code-fence : confirmer ou compléter `stripFence` (scribeResponse.js:88) pour le reliquat ~2% noté dans STATE (modèle qui enveloppe le JSON dans une fence ```json).

### Claude's Discretion
- Nommage/emplacement exact du module de fixtures (probablement à côté de `scribeResponse.spec.js`).
- Forme précise du message correctif du re-ask (texte du nudge).
- Structure exacte du helper partagé (signature, où il vit dans scribeAI.js).
</decisions>
