**Gathered:** 2026-03-22

<domain>
## Phase Boundary

This phase prepares and submits a PR to OnlyOffice's sdkjs repo adding `ApiRun.prototype.GetInlineDrawings()`, documents the build/deployment process for our patched SDK, and ensures the dev environment setup is automated and version-checked.

**Why this exists:** During Phase 23, we discovered that the OO plugin API provides no way to determine the position of inline drawings (images) within a text run. `GetElement(i)` only returns runs/hyperlinks; `GetAllDrawingObjects()` returns drawings but without position info. This blocks accurate inline image marker placement in the markdown extraction pipeline.

**What was built (already done, needs cleanup):**
- `ApiRun.prototype.GetInlineDrawings()` implementation in `word/apiBuilder.js`
- Build pipeline via `Dockerfile.build` (Closure Compiler in Docker)
- Volume mounting in `oo-dev-setup.sh`
- QUnit tests (written for v8.2, need updating for v9.3)
- Proposal document `oo-api-proposal.md`

</domain>

<decisions>
## Implementation Decisions

### API Design
- Method on ApiRun (not ApiParagraph) ŌĆö purely additive, no breaking changes
- Returns `Array<{drawing: ApiDrawing, position: number}>` ŌĆö position matches GetText() character index
- Uses `GetApiDrawing()` factory (not `new ApiDrawing()`) for proper type dispatch (ApiImage, ApiShape, etc.)
- Exported via string key `ApiRun.prototype["GetInlineDrawings"]` for Closure Compiler compatibility

### Build & Deployment
- SDK built from official ONLYOFFICE/sdkjs repo at tag v9.3.0.138 (must match container version)
- Closure Compiler via Docker (`Dockerfile.build`) ŌĆö no Java needed on host
- Compiled files mounted as Docker volumes over originals in OO container
- Plugin uses `apiRun.GetInlineDrawings()` with graceful fallback if method unavailable

- Include QUnit tests and JSDoc example file

<specifics>
## Specific Ideas

### Internal model details (from source analysis)
- Inline drawings are in `ParaRun.Content[]`, not `Paragraph.Content[]`
- Type constants: `para_Drawing=0x0016`, `para_Text=0x0001`, `para_Space=0x0002`, `para_Tab=0x0015`, `para_NewLine=0x0010`
- `GetApiDrawing()` factory at line 31071 dispatches to ApiImage/ApiShape/ApiGroup etc.
- `ApiDrawing.GetName()` calls `this.Drawing.getObjectName()`
- `GetAllDrawingObjects()` at line 11127 uses `AscBuilder.GetApiDrawings()` (not `new ApiDrawing()`)

### Version constraints
- Container OO version: 9.3.0-138
- SDK tag: v9.3.0.138 on github.com/ONLYOFFICE/sdkjs
- Closure Compiler version in package.json: `^20230228.0.0`

### Exploration history (approaches tried and failed)

Extensive exploration was done before arriving at the PR approach. Key failed attempts:

1. **API-level approaches (all failed on OO 9.3.0):**
   - `GetElement(i)` never returns drawings ŌĆö they are inside runs, not at paragraph level
   - `para.GetElements()` (suggested by OO experts) does not exist
   - `drawing.GetParentRun()` does not exist
   - `drawing.GetRange()` / `drawing.GetPosition()` do not exist
   - `drawing.GetParentParagraph()` exists but gives no position info

2. **Accessing internal model from plugin (all failed):**
   - `para.private`, `para.Paragraph`, `para.Content` ŌĆö none exist on the ApiParagraph wrapper
   - `Object.keys(para)` crashes the callCommand silently
   - Direct property probing (`para.ya`, `para.ab`, etc.) ŌĆö properties exist but none leads to a Content array
   - `AscBuilder.ApiRun` ŌĆö AscBuilder exists but constructor keys are minified

3. **Heuristic approach (partially worked):**
   - Empty runs (text="") often correspond to drawing anchors
   - Works when no whitespace is adjacent to the drawing
   - **Fails** when a space follows the image ŌĆö the anchor run merges with the space run and becomes indistinguishable
   - Fallback: place markers after last text content (approximate, not precise)

4. **Non-compiled SDK injection (failed):**
   - Concatenating source files without Closure Compiler breaks cross-chunk references
   - The two chunks (sdk-all-min.js and sdk-all.js) share a compilation scope via `--chunk`
   - Both non-minified concatenated and IIFE-wrapped versions caused `ReferenceError` or `TypeError`

5. **Building from wrong source version (failed):**
   - Initially cloned `nicedoc/sdkjs` fork (v8.2.0) ŌĆö incompatible with container v9.3.0
   - Compiled SDK from v8.2.0 caused `TypeError: Cannot read properties of undefined (reading 'arabic')` and other errors
   - Resolution: must use official `ONLYOFFICE/sdkjs` repo at exact tag `v9.3.0.138`

### Critical implementation pitfalls discovered

1. **`new ApiDrawing()` vs `GetApiDrawing()` factory:** In v9.3.0, `GetAllDrawingObjects()` switched from `new ApiDrawing(graphicObj)` to `GetApiDrawing(graphicObj)` which dispatches to ApiImage/ApiShape/etc. Using the base constructor directly produces objects where `GetName()` returns `undefined` because type-specific wiring is missing.

2. **Image serialization: Copy() vs ToJSON():** Initially used `drawing.ToJSON(false, false)` to serialize images before `InsertContent` destroys the selection. ToJSON preserves dimensions but **loses the bitmap data** ŌĆö images appear as white rectangles with correct size. `drawing.Copy()` is the correct approach ŌĆö it creates a deep in-memory copy that preserves the actual image content.

3. **Image naming: SetName() condition:** The original extraction code checked `if (!name || name === "")` before calling `SetName("scribe-img-N")`. But OO drawings always have a default name like `Picture 1427054139`. The condition was changed to `if (!name || name.indexOf("scribe-img-") !== 0)` to always rename drawings that don't already have a scribe name.
